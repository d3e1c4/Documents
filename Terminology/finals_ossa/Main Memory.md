### **Main Memory යනු කුමක්ද? (Definition)**

**Main Memory** යනු එකිනෙකට වෙනස් වූ ලිපින (**addresses**) සහිත **words** හෝ **bytes** විශාල සමූහයකින් සැදුම්ලත් ගබඩාවකි. මෙය **CPU** සහ **I/O devices** මඟින් හවුලේ භාවිතා කරන, ඉතා ඉක්මනින් ප්‍රවේශ විය හැකි දත්ත තැන්පත් කර තබන ස්ථානයක් (**repository**) ලෙස හැඳින්වේ.

---

### **Main Memory සංකල්පය (The Concept)**

**Main Memory** එකක ක්‍රියාකාරීත්වය සහ එහි වැදගත්කම අවබෝධ කර ගැනීමට පහත කරුණු උපකාරී වේ:

- **CPU සෘජු ප්‍රවේශය (Direct Access):** **CPU** එකට සෘජුවම සම්බන්ධ වී දත්ත ලබාගත හැක්කේ **Main Memory** සහ එහි ඇති **registers** වලින් පමණි.
    
- **දත්තවල ස්වභාවය (Volatile Storage):** මෙය තාවකාලික මතකයකි. පද්ධතියේ දෝෂයක් (**system failure**) සිදු වූ විට එහි ඇති සියලුම දත්ත මැකී යයි.
    
- **වැඩසටහන් ක්‍රියාත්මක කිරීම (Program Execution):** ඕනෑම වැඩසටහනක් ක්‍රියාත්මක වීමට පෙර, එය **Hard Disk (Secondary Storage)** එකේ සිට **Main Memory** වෙත පූරණය කළ යුතුය.
    
- **මතකය බෙදා වෙන් කිරීම (Memory Partitioning):** සාමාන්‍යයෙන් **Main Memory** කොටස් දෙකකට බෙදා ඇත:
    
    - **Kernel memory area:** මෙහි **Operating System**, **PCBs** සහ **System calls** වැනි වැදගත් දත්ත ගබඩා වේ.
        
    - **User memory area:** මෙහි පරිශීලකයා විසින් ක්‍රියාත්මක කරන විවිධ **processes** පවතියි.
        
- **ලිපින පරාසය (Memory Capacity):** මතකයේ ඇති ලිපින සඳහා $n$ bits ප්‍රමාණයක් භාවිතා කරන්නේ නම්, එම මතකයේ ධාරිතාව $2^{n}$ bytes වේ.

**Virtual Memory** කියන්නේ ඇත්තටම "මැජික්" එකක් වගේ දෙයක්. ඒකෙන් කරන්නේ පරිගණකය පාවිච්චි කරන කෙනාව සහ වැඩසටහන් (Programs) රවට්ටන එකයි!

### 1. ගැටලුව (The Problem)

අපි දන්නවා ඕනෑම වැඩසටහනක් (Process) Run වෙන්න නම් ඒක **RAM (Physical Memory)** එකට පටවන්න (Load) ඕනේ කියලා.

- හිතන්න ඔයාගේ පරිගණකයේ තියෙන්නේ **4GB RAM** එකක් විතරයි.
    
- හැබැයි ඔයාට Run කරන්න ඕනේ **10GB** සයිස් එකේ අලුත් වීඩියෝ ගේම් එකක්!
    
- සාමාන්‍ය විදිහට බැලුවොත් මේක කරන්න බැහැ නේද? මොකද 10GB ගේම් එකක් 4GB RAM එකකට දාන්න බැහැ.
    

### 2. විසඳුම - Virtual Memory (The Magic)

මෙන්න මෙතනදී තමයි Operating System එක **Virtual Memory** කියන සංකල්පය පාවිච්චි කරන්නේ.

- **Logical Memory (හිතාගත් මතකය):** OS එක අර වීඩියෝ ගේම් එකට කියනවා, "බය වෙන්න එපා, මං ගාව ඔයාට ඕන තරම් ලොකු RAM එකක් තියෙනවා. ඔයාගේ ඔක්කොම දත්ත මට දෙන්න" කියලා. ඒ කියන්නේ ගේම් එක හිතාගෙන ඉන්නේ එයාට 10GB තරම් ලොකු RAM එකක් හම්බුණා කියලයි.
    
- **Physical Memory (සැබෑ RAM එක) සහ Backing Store (Hard Disk):** හැබැයි OS එක හොරෙන් කරන්නේ, අර 10GB ගේම් එකෙන් **දැනට ඇත්තටම පාවිච්චි වෙන** පොඩි කොටසක් විතරක් (උදාහරණයක් විදිහට 2GB වගේ) ඇත්ත RAM එකේ තියාගන්නවා. ඉතිරි 8GB ම තියාගන්නේ **Hard Disk** එකෙයි.
    

### 3. කලින් පාඩම් එක්ක සම්බන්ධය (The Connection)

අපි කලින් කතා කරපු දේවල් මේකට සම්බන්ධ වෙන්නේ මෙහෙමයි:

1. **Demand Paging:** අර 10GB ගේම් එකෙන් මුලින්ම ඕන වෙන කොටස විතරක් RAM එකට ගේන ක්‍රමයට තමයි Demand Paging කිව්වේ.
    
2. **Page Fault:** ඔයා ගේම් එකේ අලුත් Level එකකට යද්දී, ඒ දත්ත RAM එකේ නැතුව Hard Disk එකේ තිබුණොත්, අන්න ඒ වෙලාවට OS එකට "ට්‍රැප්" එකක් යනවා. ඒක තමයි Page Fault එක.
    
3. **Page Replacement (LRU, FIFO):** දැන් ඒ අලුත් Level එක RAM එකට ගේන්න බලද්දී RAM එක (අර 4GB) පිරිලා නම්, OS එක කරන්නේ පරණ Level එකක දත්ත RAM එකෙන් මකලා (Victim), ඒ ඉඩට අලුත් Level එකේ දත්ත ගේන එකයි.
    
4. **Page Buffering:** මේ අයින් කරන සහ ගේන ක්‍රියාවලිය (Swapping) ඉක්මන් කරන්න තමයි අපි අන්තිමට කතා කරපු "Free Frame Pool" වගේ දේවල් පාවිච්චි කරන්නේ.


> "**Virtual Memory** කියන්නේ, පරිගණකයේ තියෙන සැබෑ **RAM (Physical Memory)** එකේ ඉඩකඩ මදි වන විට, **Hard Disk** (හෝ **SSD**) එකේ කොටසක් තාවකාලිකව **RAM** එක විදිහට පාවිච්චි කිරීමට **Operating System** එක මඟින් යොදා ගන්නා තාක්ෂණික උපක්‍රමයයි."


RAM Memory Addressing සහ ඒ ආශ්‍රිත මතක ධාරිතාව (Memory Size) ගණනය කරන ආකාරය ගැන අපි මුල සිට සරලව සාකච්ඡා කරමු.

### 1. RAM එකක් යනු කුමක්ද? (සරල උදාහරණයක්)

RAM එකක් කියන්නේ කුඩා ලොකර් (Lockers) විශාල ප්‍රමාණයක් තියෙන එකතුවක් වගෙයි. මේ සෑම ලොකර් එකකම අපිට දත්ත 1 **Byte** එකක් ගබඩා කරන්න පුළුවන්.

- **Address:** මේ ලොකර් එකිනෙකින් වෙන් කර හඳුනාගන්න ඒ සෑම එකකටම අංකයක් දීලා තියෙනවා. ඒකට අපි කියන්නේ **Address** එක කියලා.
    
- **CPU Interaction:** CPU එකට යම් දත්තයක් අවශ්‍ය වුණාම, ඒ දත්තය තියෙන්නේ කීවෙනි ලොකර් එකේද (Address) කියලා දැනගෙන ඒ අදාළ ස්ථානයට යනවා.

### 2. Bits ගණන සහ මතකයේ ප්‍රමාණය (The $2^n$ Rule)

පරිගණකය දන්නේ **Binary** (0 සහ 1) විතරයි. ඒ නිසා මේ ලොකර් වලට අංක දෙන්නෙත් Binary වලින්. අපිට ලැබෙන **Bits** ගණන (n) අනුව අපිට කීවෙනි අංකය වෙනකම් ලොකර් වලට නම් දෙන්න පුළුවන්ද කියලා තීරණය වෙනවා.

සූත්‍රය: **Total Addresses = $2^n$**

ඔයා දීපු **16-bit** උදාහරණය ගත්තොත්:

- මෙහි $n = 16$ කි.
    
- එවිට ලිපින ගණන = $2^{16} = 65,536$ කි.
    
- මෙයින් අදහස් කරන්නේ ලිපින (ලිපිනය 0 සිට 65,535 දක්වා) 65,536 ක් ඇති බවයි.
    
- සෑම ලිපිනයකම 1 **Byte** බැගින් ඇති නිසා, මුළු මතකය = **65,536 Bytes**.
    

---

### 3. Size එක ගණනය කරන්නේ කොහොමද?

අපි සාමාන්‍යයෙන් මතකය කියන්නේ **Bytes** වලින් නෙවෙයි, **KB**, **MB** හෝ **GB** වලින්.

- $1024\ \text{Bytes} = 1\ \text{KB}$ (Kilobyte)
    
- එවිට, $65,536\ \text{Bytes} = \frac{65,536}{1024} = \mathbf{64\ \text{KB}}$
    

---

### 4. Hexadecimal (Hex) භාවිතා කරන්නේ ඇයි?

Binary වලින් ලිපින ලියනකොට ඒක ඉතා දිගු වෙනවා (උදා: `1111 1111 1111 1111`). මේක ලියන්න සහ කියවන්න අපහසු නිසා අපි **Hexadecimal** භාවිතා කරනවා.

- **Binary bits 4ක්** එක **Hex** අංකයකින් පෙන්වන්න පුළුවන්.
    
- ඒ නිසා **16-bit Binary** අංකයක්, **Hex** අංක 4කින් (`FFFF`) කෙටියෙන් ලියන්න පුළුවන්.
    

---

### 5. Modern Systems වල තත්ත්වය

අද කාලයේ පරිගණක වල තියෙන **Address Bus** එක 16-bit වලට වඩා ගොඩක් වැඩියි.

- **32-bit System:** $2^{32} \approx 4,294,967,296\ \text{Bytes} \approx \mathbf{4\ \text{GB}}$. ඒ නිසයි 32-bit OS එකක 4GB වලට වඩා RAM එකක් පාවිච්චි කරන්න බැරි.
    
- **64-bit System:** මෙහි $2^{64}$ යනු ඉතා විශාල අගයක්. එමගින් අපිට **Terabytes** ගණනාවක RAM එකක් වුවද **Address** කිරීමේ හැකියාව ලැබෙනවා.

ඔයා අහපු ප්‍රශ්නය හරිම වටිනවා. ගොඩක් අයට පැටලෙන තැනක් තමයි මේ **Byte** එකේ ප්‍රමාණය සහ **Address** එකේ ප්‍රමාණය. අපි මේක සරලව වෙන් කරලා හඳුනා ගනිමු.

### 1. 1 Byte = 8 bits වුණත් ඇයි 16-bit (හෝ 32/64) කියලා ගන්නේ?

මෙහිදී ඔයා පටලවාගෙන තියෙන්නේ කරුණු දෙකක්:

- **Data Size (දත්තයේ ප්‍රමාණය):** මතකයේ ඇති එක "කොටුවක" (Cell) ගබඩා කළ හැකි ප්‍රමාණය තමයි 1 **Byte** (8 **bits**) කියන්නේ. මේක ලෝක සම්මත මිනුමක්.
    
- **Address Size (ලිපිනයේ ප්‍රමාණය):** 16-bit, 32-bit හෝ 64-bit කියන්නේ ඒ කොටු වලට අංක (Addresses) දෙන්න පාවිච්චි කරන **Bits** ගණනයි.
    

**සරල උදාහරණයක්:**

හිතන්න ඔයා ළඟ තැපැල් පෙට්ටි (**RAM Cells**) දහස් ගණනක් තියෙනවා කියලා.

- එක පෙට්ටියක් ඇතුළට දාන්න පුළුවන් ලියුම් ප්‍රමාණය (Capacity) = 8 **bits** (1 **Byte**).
    
- හැබැයි මේ පෙට්ටි දහස් ගණනක් තියෙන නිසා, ඒ හැම එකකටම වෙන වෙනම අංක (**Address**) දෙන්න ඕනේ. ඒ අංකය ලියන්න ඔයාට ඉලක්කම් කිහිපයක් ඕනේ වෙනවා.
    
- අන්න ඒ ලිපිනය ලියන්න පාවිච්චි කරන දිග තමයි 16-bit කියන්නේ. 16-bit ලිපිනයක් පාවිච්චි කළොත් ඔයාට පෙට්ටි 65,536 කට ලිපින දෙන්න පුළුවන් ($2^{16} = 65,536$).
    

---

### 2. RAM එකක දත්ත ගබඩා වෙන්නේ කොහොමද?

RAM එකක් කියන්නේ කුඩා විද්‍යුත් කොටස් (Electronic Components) මිලියන ගණනක එකතුවක්.

**පියවරෙන් පියවර ක්‍රියාවලිය:**

1. **Memory Grid:** RAM එක ඇතුළේ දත්ත ගබඩා වෙන්නේ පේළි (**Rows**) සහ තීරු (**Columns**) සහිත දැලක් (Grid) වැනි සැකැස්මකයි. මෙහි ඇති සෑම හමුවන ලක්ෂ්‍යයක්ම එක **Bit** එකක් ගබඩා කරන කුඩා "කොටුවක්" වගෙයි.
    
2. **Addressing:** CPU එකට යම් දත්තයක් ගබඩා කරන්න ඕන වුණාම, මුලින්ම ඒ දත්තය දාන්න ඕනේ "කොටුවේ" ලිපිනය (**Address**) **Address Bus** එක හරහා එවනවා.
    
3. **Opening the cell:** ඒ ලිපිනයට අදාළව ඇති විද්‍යුත් ස්විචයන් (Transistors) ක්‍රියාත්මක වෙලා, අදාළ පේළිය සහ තීරුව විවෘත කරනවා.
    
4. **Storing Data:** දැන් **Data Bus** එක හරහා එන විද්‍යුත් සංඥා (0 සහ 1) ඒ කොටු ඇතුළට යනවා.
    
    - විදුලිය ගබඩා වුණොත් (**Charge**) = 1 (Bit).
        
    - විදුලිය නැති වුණොත් (**No charge**) = 0 (Bit).
        

**මතක තබා ගත යුතු වැදගත්ම දේ:**

RAM එක යනු **Volatile memory** එකකි. එනම්, පරිගණකයේ විදුලිය විසන්ධි වූ සැනින් මේ කොටු වල තිබූ විද්‍යුත් ආරෝපණ (Charges) නැති වී යන අතර ඒ සමඟම දත්ත ද මැකී යයි.


**Address Binding**
වැඩසටහනක් (Program) ලියද්දී අපි ලිපිනයන් විදියට පාවිච්චි කරන්නේ විචල්‍යයන් (Variables - උදා: `x`, `count`). හැබැයි මේ වැඩසටහන RAM එකේ Run වෙන්න නම්, ඒ විචල්‍යයන් වලට RAM එකේ සැබෑ භෞතික ලිපිනයක් (**Physical Address**) ලැබෙන්න ඕනේ. මෙන්න මේ "මවාගත් ලිපිනය" "සැබෑ ලිපිනයට" සම්බන්ධ කරන ක්‍රියාවලිය තමයි **Address Binding** කියන්නේ.

**Compile Time Address Binding යනු කුමක්ද?**

වැඩසටහන Compile කරන වෙලාවේදීම (ඒ කියන්නේ Source code එක Binary බවට හරවන වෙලාවේදීම), එම වැඩසටහන RAM එකේ පිහිටිය යුතු නිවැරදි ස්ථානය කලින්ම දන්නවා නම්, Compiler එක විසින් කෙලින්ම **Absolute Addresses** (සෘජු ලිපින) සහිත කෝඩ් එකක් ජනනය කරනවා.


----------------------------------------------

## Swapping යනු කුමක්ද?

සරලවම කිව්වොත්, **Swapping** කියන්නේ මතකයේ (**RAM**) ඇති ඉඩ ප්‍රමාණය මදි වූ විට, දැනට ක්‍රියාත්මක නොවන වැඩසටහනක් (**Process**) තාවකාලිකව **RAM** එකෙන් ඉවත් කර **Hard Disk** එකට යැවීම සහ පසුව නැවත අවශ්‍ය වූ විට එය **RAM** එකට ගෙන ඒමයි.

මෙය ප්‍රධාන පියවර දෙකකින් සිද්ධ වෙනවා:

1. **Swap Out:** වැඩසටහනක් (Process) **Main Memory (RAM)** එකෙන් ඉවත් කර **Hard Disk** (Secondary Storage) එකේ ඇති **Backing Store** කියන කොටසට යැවීම.
    
2. **Swap In:** පසුව එම වැඩසටහනට නැවත **CPU** එක අවශ්‍ය වූ විට, එය **Hard Disk** එකේ සිට නැවත **Main Memory** එකට ගෙන ඒම.
    

---

### ඇයි Swapping වැදගත් වෙන්නේ?

- **Multiprogramming:** ඔයාගේ **RAM** එකේ ඉඩ තියෙන ප්‍රමාණයට වඩා වැඩි වැඩසටහන් ප්‍රමාණයක් එකවර ධාවනය කිරීමට මින් අවස්ථාව ලැබෙනවා.
    
- **Priority (ප්‍රමුඛතාවය):** වැඩි ප්‍රමුඛතාවයක් ඇති (**High priority**) වැඩසටහනක් ආ විට, අඩු ප්‍රමුඛතාවයක් ඇති වැඩසටහනක් **Swap Out** කර ඉඩ ලබා දීමට **OS** එකට පුළුවන්. මෙයට **Roll out, Roll in** කියලත් කියනවා.

### සරල උදාහරණයක්

හිතන්න ඔයාගේ ලඟ තියෙනවා **වැඩ කරන මේසයක්** (RAM) සහ **ලොකු අල්මාරියක්** (Hard Disk).

- ඔයා මේසය උඩ පොත් 5ක් තියාගෙන වැඩ කරනවා. දැන් 6 වෙනි පොතකුත් ඕනේ වෙනවා, හැබැයි මේසය උඩ ඉඩ නැහැ.
    
- එතකොට ඔයා කරන්නේ මේසය උඩ තියෙන, දැන් පාවිච්චි නොකරන පොතක් අරන් අල්මාරියට දාන එකයි (**Swap Out**).
    
- දැන් මේසය උඩ ඉඩ තියෙන නිසා අලුත් පොත අරන් වැඩ කරන්න පුළුවන්.
    
- ආයෙත් අර පරණ පොත ඕන වුණාම, අල්මාරියෙන් අරන් මේසයට ගේනවා (**Swap In**).
    

මෙන්න මේ ක්‍රියාවලිය තමයි **Swapping** කියන්නේ. මෙය නිසා තමයි ඔයාගේ පරිගණකයේ **RAM** එක 8GB වුණත්, ඊට වඩා ලොකු වැඩ ප්‍රමාණයක් කරගන්න පුළුවන් වෙන්නේ


## Contiguous Allocation යනු කුමක්ද?

මෙහිදී සිදුවන්නේ සෑම වැඩසටහනකටම (**Process**) **RAM** එකෙහි **එක පෙළට පිහිටි, නොකැඩුණු** තනි මතක කොටසක් (**Single continuous block**) වෙන් කර දීමයි.

වැඩසටහනක් මතකයට පටවන විට, **Operating System** එක විසින් එම වැඩසටහන සම්පූර්ණයෙන්ම ඇතුළත් කළ හැකි තරමේ විශාල, එක දිගට පවතින ඉඩක් (**Contiguous block**) සොයා ගනු ලබනවා.

---

### ආරක්ෂාව සඳහා Registers (Address Protection)

එක වැඩසටහනකට තවත් වැඩසටහනක මතකයට හොරෙන් ඇතුළු වීමට නොහැකි වන සේ ආරක්ෂාව සලසන්නේ මෙන්න මේ **Registers** දෙක මගින්:

1. **Base Register:** වැඩසටහනට වෙන් කළ මතක කොටසේ ඇති **කුඩාම ලිපිනය** (ආරම්භක ලිපිනය) මෙහි තබා ගනියි.
    
2. **Limit Register:** එම වැඩසටහනට ලබා දී ඇති **මතක පරාසයේ ප්‍රමාණය** (Size) මෙහි සටහන් වේ.

### මෘදුකාංග (Software) ඇසුරින් උදාහරණයක්

හිතන්න ඔයාගේ **RAM** එක පේළියකට තියෙන **Parking Slot** (වාහන නවත්වන ස්ථාන) පෙළක් කියලා.

- **VLC Player (200MB):** ඔයා VLC එක Open කරනවා. OS එක බලනවා 200MB ම එක දිගට නවත්වන්න පුළුවන් තැනක් කොහෙද තියෙන්නේ කියලා. හිතන්න 1000 වෙනි ස්ලොට් එකේ ඉඳන් ඉඩ තියෙනවා කියලා. එතකොට VLC එකට 1000 සිට 1199 දක්වා ඉඩ දෙනවා. (මෙහි **Base** = 1000, **Limit** = 200).
    
- **Notepad (50MB):** ඊළඟට ඔයා නොට්පෑඩ් එක ඇරියම, OS එක ඒකට 1200 සිට 1249 දක්වා ඉඩ දෙනවා.
    
- **Calculator (20MB):** ඒකට 1250 සිට 1269 දක්වා ඉඩ දෙනවා.
    

**මෙහි ඇති ගැටලුව (The Downside):** දැන් හිතන්න ඔයා මැද තියෙන **Notepad** එක Close කරනවා කියලා. එතකොට 1200 ඉඳන් 1249 වෙනකම් (50MB) ඉඩක් හැදෙනවා. හැබැයි ඔයාට දැන් 60MB තියෙන අලුත් **Game** එකක් Open කරන්න ඕන වුණොත්, අර 50MB කොටස ඒකට මදි. ඒ වගේම ඒක "කෑලි වලට කඩලා" දාන්නත් බැහැ, මොකද මේක **Contiguous** (එක දිගට තියෙන්න ඕන) ක්‍රමය නිසයි.

මෙන්න මේ නිසා මතකයේ ඉඩ තිබුණත් ඒවා පාවිච්චි කරන්න බැරි වෙනවා. මේකට තමයි අපි **Fragmentation** කියලා කියන්නේ

**Multiple-Partition Allocation** කියන්නේ එකම වෙලාවේ **Processes** කිහිපයකට **RAM** එකේ ඉඩ වෙන් කර දෙන ක්‍රමයක්. අපි මේක සරලව කොටස් දෙකකට බෙදාගෙන තේරුම් ගනිමු.

මෙහිදී ප්‍රධාන ක්‍රම දෙකක් තියෙනවා:

### 1. Fixed Partitioning (ස්ථාවර ලෙස කොටස් කිරීම)

මෙහිදී **Operating System** එක විසින් **RAM** එක කලින්ම නිශ්චිත ප්‍රමාණයේ කොටස් (**Partitions**) වලට බෙදා තබනවා.

- එක් කොටසක ක්‍රියාත්මක කළ හැක්කේ එක **Process** එකක් පමණයි.
    
- **ගැටලුව:** වැඩසටහනක් තමන්ට ලැබුණු කොටසට වඩා කුඩා නම්, ඉතිරි ඉඩ ප්‍රමාණය අපතේ යනවා. මෙයට **Internal Fragmentation** කියලා කියනවා.
    

### 2. Variable/Dynamic Partitioning (විචල්‍ය ලෙස කොටස් කිරීම)

මෙහිදී **RAM** එක කලින් බෙදා තබන්නේ නැහැ. අලුත් වැඩසටහනක් එන විට, ඒ මොහොතේ ඒ වැඩසටහනට අවශ්‍ය ප්‍රමාණයටම **Partition** එකක් වෙන් කර දෙනවා.

- මෙහිදී ඉතිරි වන හිස් මතක කොටස් වලට අපි **Hole** එකක් කියලා කියනවා.
    
- **ගැටලුව:** වැඩසටහන් අවසන් වී ඉවත් වන විට, මතකයේ තැනින් තැන කුඩා ඉඩවල් (Holes) ඉතිරි වෙනවා. මේවා එකතු කළ විට විශාල ඉඩක් තිබුණත්, ඒවා එක දිගට (**Contiguous**) නොමැති නිසා අලුත් වැඩසටහනකට භාවිතා කළ නොහැකි වෙනවා. මෙයට **External Fragmentation** කියලා කියනවා.
    

---

### මෘදුකාංග (Software) ඇසුරින් උදාහරණයක්

හිතන්න ඔයාගේ **RAM** එක **ලොකු හෝටලයක්** වගේ කියලා:

- **Fixed Partitioning:** හෝටලයේ තියෙන්නේ ඇඳන් 4ක් තියෙන ලොකු කාමර විතරයි. දැන් **Calculator** වගේ පොඩි වැඩසටහනක් (Single person) ආවත් එයාට දෙන්නේ අර ඇඳන් 4 කාමරයමයි. ඉතිරි ඇඳන් 3ම අපතේ යනවා (**Internal Fragmentation**).
    
- **Variable Partitioning:** කවුරු හරි එන වෙලාවට එයාට ඕන සයිස් එකටම කාමරයක් වෙන් කරලා දෙනවා.
    
    - මුලින් **Photoshop (4GB)** එනවා, එයාට ඒ ගාණ දෙනවා.
        
    - ඊළඟට **VLC (1GB)** එනවා, ඒකටත් දෙනවා.
        
    - දැන් මැද ඉන්න **VLC** අයින් වුණොත්, එතන 1GB හිස් ඉඩක් (**Hole**) හැදෙනවා.
        
    - හැබැයි ඊළඟට එන වැඩසටහනට 1.5GB ඕන නම්, අර 1GB වලට ඒක දාන්න බැහැ. හෝටලේ තව ඉඩ තිබුණත් ඒක කෑලි වලට තියෙන නිසා පාවිච්චි කරන්න බැහැ (**External Fragmentation**).
        

---

### දත්ත ගබඩා කිරීමට සුදුසු "Hole" එකක් තෝරාගන්නා ක්‍රම (Dynamic Storage-Allocation)

හිස් ඉඩවල් (Holes) කිහිපයක් ඇති විට, අලුත් වැඩසටහනකට ඉඩ දිය යුතු ආකාරය තීරණය කරන ක්‍රම 3ක් තියෙනවා:

1. **First-fit:** ඉඩ ඇති පළමු සිදුරටම (**First hole**) වැඩසටහන ඇතුළත් කිරීම. මෙය වේගවත් ක්‍රමයකි.
    
2. **Best-fit:** වැඩසටහනේ ප්‍රමාණයට වඩාත්ම ගැළපෙන, කුඩාම සිදුර තෝරා ගැනීම. මෙහිදී ඉතා කුඩා ඉතිරි ඉඩවල් ඉතිරි වේ.
    
3. **Worst-fit:** මතකයේ ඇති විශාලතම සිදුර තෝරා ගැනීම. මෙහි අදහස වන්නේ ඉතිරි වන කොටස තවත් වැඩසටහනකට පාවිච්චි කළ හැකි තරම් විශාල වනු ඇති බවයි.

**Dynamic Storage-Allocation Problem** කියන්නේ මතකයේ (RAM) තැනින් තැන පවතින හිස් ඉඩවල් (**Holes**) අතරින්, අලුතින් එන වැඩසටහනකට (**Process**) වඩාත්ම ගැළපෙන ඉඩක් තෝරාගන්නේ කොහොමද කියන ගැටලුවයි.

අපි හිතමු ඔයාගේ **RAM** එකේ දැනට මෙන්න මේ වගේ හිස් ඉඩවල් (**Holes**) පිළිවෙළට තියෙනවා කියලා:

1. 100 KB
    
2. **500 KB**
    
3. 200 KB
    
4. **300 KB**
    
5. **600 KB**
    

දැන් අලුතින් **212 KB** ප්‍රමාණයේ වැඩසටහනක් (**Process**) එනවා. මේ ක්‍රම 3න් ඒකට ඉඩ වෙන් කරන්නේ මෙහෙමයි:

---

### 1. First-fit (පළමු ගැළපීම)

නමට අනුවම, ලිස්ට් එකේ මුල ඉඳන් බලද්දී හමුවෙන **පළමු සුදුසු ඉඩට (Hole)** වැඩසටහන ඇතුළත් කරනවා.

- **උදාහරණය:** 100KB (මදි), ඊළඟට තියෙන්නේ 500KB. ඒක 212KB ට වඩා ලොකු නිසා එසැනින් වැඩසටහන **500 KB** සිදුරට ඇතුළත් කරනවා.
    
- **වාසිය:** හරිම වේගවත් (**Speed**), මොකක්ද මුළු ලිස්ට් එකම පීරන්න ඕන නැති නිසා.
    

### 2. Best-fit (හොඳම ගැළපීම)

වැඩසටහනේ ප්‍රමාණයට වඩාත්ම ආසන්න (ඒ කියන්නේ ඉඩ ඇති සිදුරු අතරින් **කුඩාම සිදුර**) තෝරාගන්නවා. මේ සඳහා මුළු ලිස්ට් එකම පරීක්ෂා කළ යුතුයි (**Search entire list**).

- **උදාහරණය:** 500, 300 සහ 600 යන සිදුරු තුනම 212ට වඩා ලොකුයි. ඒ අතරින් කුඩාම එක වන්නේ **300 KB** සිදුරයි. ඒ නිසා වැඩසටහන මෙතනට දානවා.
    
- **විශේෂත්වය:** මෙහිදී ඉතිරි වන ඉඩ ප්‍රමාණය (**Leftover**) ඉතා කුඩායි.
    

### 3. Worst-fit (නරකම ගැළපීම)

මතකයේ ඇති **විශාලතම සිදුර (Largest hole)** තෝරාගන්නවා. මෙහිදීත් මුළු ලිස්ට් එකම පරීක්ෂා කරන්න වෙනවා.

- **උදාහරණය:** තියෙන සිදුරු වලින් විශාලතම එක **600 KB** නිසා වැඩසටහන එතනට ඇතුළත් කරනවා.
    
- **අරමුණ:** මෙහි අදහස වන්නේ ඉතිරි වන ඉඩ කොටස (**Leftover**) තවත් ලොකු වැඩසටහනකට පාවිච්චි කරන්න පුළුවන් තරම් ලොකුවට ඉතුරු වේවි කියලයි.

### සංසන්දනය (Comparison)

- **First-fit සහ Best-fit** යන ක්‍රම දෙකම **Worst-fit** වලට වඩා වේගය (**Speed**) සහ මතකය පාවිච්චි කිරීමේ හැකියාව (**Storage utilization**) අතින් උසස් කියලා සැලකෙනවා.
    
- **First-fit** සාමාන්‍යයෙන් වඩාත්ම වේගවත් ක්‍රමයයි.
    

**සරල උදාහරණයක්:** හිතන්න ඔයා වාහනයක් පාක් කරන්න යනවා කියලා.

- **First-fit:** හමුවෙන පළමු හිස් ඉඩේ පාක් කරනවා.
    
- **Best-fit:** හමුවෙන හැම පාර්කින් ස්ලොට් එකක්ම බලලා, වාහනේට හරියටම තද වෙන්න සෙට් වෙන පොඩිම ඉඩ බලලා පාක් කරනවා.
    
- **Worst-fit:** තියෙන ලොකුම ඉඩ තියෙන තැන බලලා පාක් කරනවා.

**Fragmentation** කියන්නේ Memory Management වලදී මතකය (RAM) අපතේ යන එක්තරා අවස්ථාවක්. සරලවම කිව්වොත්, මතකයේ ඉඩ තිබුණත් එය නිවැරදිව පාවිච්චි කිරීමට නොහැකි වීමයි.

මෙය ප්‍රධාන වශයෙන් ආකාර දෙකකි:

### 1. External Fragmentation (බාහිර ඛණ්ඩනය)

මෙහිදී සිදුවන්නේ මතකයේ තැනින් තැන හිස් ඉඩවල් (**Holes**) විසිරී පැවතීමයි. පද්ධතියේ මුළු හිස් ඉඩ ප්‍රමාණය ගත්තොත් එය අලුත් වැඩසටහනකට සෑහෙන නමුත්, එම ඉඩ **එක දිගට (Contiguous)** නොමැති නිසා එය භාවිතා කළ නොහැකි වේ.

- **මෘදුකාංග උදාහරණයක්:** හිතන්න ඔයාගේ RAM එකේ තැනින් තැන 20MB, 30MB සහ 50MB ලෙස හිස් ඉඩවල් (Holes) 3ක් තියෙනවා කියලා. මුළු එකතුව 100MB. දැන් ඔයාට 80MB ප්‍රමාණයේ **VLC Player** එකක් Run කරන්න ඕනේ. මුළු ඉඩ 100MB තිබුණත්, එක දිගට 80MB නැති නිසා OS එකට ඒක මතකයට දාන්න බැහැ.
    
- **විසඳුම:** මෙයට විසඳුම වන්නේ **Compaction** කියන ක්‍රියාවලියයි. මෙහිදී මතකයේ ඇති සියලුම වැඩසටහන් එක පැත්තකට කර, හිස් ඉඩවල් සියල්ල එකතු කර විශාල තනි බ්ලොක් එකක් සාදනු ලබයි.
    

---

### 2. Internal Fragmentation (අභ්‍යන්තර ඛණ්ඩනය)

මෙය සිදුවන්නේ **Fixed Partitioning** (ස්ථාවර ලෙස කොටස් කළ) මතකයකයි. මෙහිදී වැඩසටහනකට ලබා දෙන මතක කොටස (**Partition**), එම වැඩසටහනේ ප්‍රමාණයට වඩා විශාල වූ විට, එම කොටස ඇතුළතම ඉතිරි වන ඉඩ අපතේ යයි.

- **මෘදුකාංග උදාහරණයක්:** හිතන්න OS එක විසින් RAM එක 100MB බැගින් වූ පෙට්ටි වලට බෙදලා තියෙනවා කියලා. දැන් ඔයා 10MB ප්‍රමාණයේ **Calculator** එකක් Open කරනවා. එතකොට ඒකට අර 100MB පෙට්ටියම ලැබෙනවා. දැන් ඒ පෙට්ටිය ඇතුළේ 90MB ම ඉතුරුයි. හැබැයි ඒ 90MB වෙන වැඩසටහනකට දෙන්නත් බැහැ, මොකද ඒ පෙට්ටිය දැනටමත් Calculator එකට අයිති නිසයි.
    

---

### සාරාංශය:

- **External:** මතකයේ තැනින් තැන ඉඩ තිබුණත් ඒවා "එක දිගට" නැති නිසා පාවිච්චි කරන්න බැහැ.
    
- **Internal:** වැඩසටහනට ලැබුණු මතක කොටස ඇතුළතම ඉඩ අපතේ යනවා.


**External Fragmentation** සහ එය විසඳීමට භාවිතා කරන **Compaction** ක්‍රියාවලිය ගැන අපි සරලව සාකච්ඡා කරමු.

### 1. External Fragmentation (බාහිර ඛණ්ඩනය)

මෙය සිදුවන්නේ මතකයේ (RAM) තැනින් තැන කුඩා හිස් ඉඩවල් (**Holes**) විසිරී පවතින අවස්ථාවකයි.

- **ගැටලුව:** පද්ධතියේ ඇති මුළු හිස් ඉඩ ප්‍රමාණය (Total free memory) එකතු කළ විට අලුත් වැඩසටහනකට සෑහෙන ප්‍රමාණයක් තිබුණත්, එම ඉඩ **එක දිගට (Contiguous)** නොමැති නිසා එම වැඩසටහන මතකයට (Load) ගැනීමට නොහැකි වේ.
    

---

### 2. Compaction (සංයුක්ත කිරීම)

**External Fragmentation** ගැටලුව විසඳීමට ඇති ප්‍රධානතම ක්‍රමය තමයි **Compaction** කියන්නේ.

- **ක්‍රියාවලිය:** මතකයේ (RAM) දැනට විසිරී පවතින වැඩසටහන් සියල්ල එක පැත්තකට තල්ලු කිරීම (Shuffle) මෙහිදී සිදු කෙරේ.
    
- **ප්‍රතිඵලය:** මෙලෙස වැඩසටහන් එකිනෙකට යාබදව තැබූ විට, විසිරී තිබූ සියලුම කුඩා හිස් ඉඩවල් එකතු වී එකම විශාල හිස් මතක කොටසක් (**One large block**) නිර්මාණය වේ.
    

---

### 3. මෙය කළ හැක්කේ කවදාද?

**Compaction** ක්‍රියාවලිය සැමවිටම කළ නොහැකියි. එය කළ හැක්කේ එක් වැදගත් කොන්දේසියක් යටතේ පමණි:

- **Dynamic Relocation:** ලිපින බැඳීම (**Address binding**) වැඩසටහන ක්‍රියාත්මක වන අවස්ථාවේදී (**Execution time**) සිදුවන්නේ නම් පමණක් මෙය කළ හැකියි.
    
- එනම්, වැඩසටහනක් මතකයේ (RAM) තිබියදීම එහි ආරම්භක ලිපිනය (**Base register**) වෙනස් කරමින් එය වෙනත් තැනකට ගෙන යාමට **OS** එකට හැකියාව තිබිය යුතුයි.
    

---

### මෘදුකාංග ඇසුරින් උදාහරණයක්

හිතන්න ඔයාගේ **RAM** එකේ මෙහෙම තියෙනවා කියලා:

1. **Chrome** (ඉහළින්ම)
    
2. **Notepad** (Close කර ඇති නිසා 50MB හිස් ඉඩක් - Hole)
    
3. **VLC Player**
    
4. තවත් 50MB හිස් ඉඩක් (Hole)
    

දැන් ඔයාට 80MB ප්‍රමාණයේ **Calculator** එකක් Open කරන්න ඕනේ. මුළු ඉඩ 100MB ($50+50$) තිබුණත්, ඒක කෑලි දෙකකට තියෙන නිසා Calculator එක මතකයට දාන්න බැහැ (**External Fragmentation**).

දැන් **OS** එක **Compaction** කරනවා:

- එය කරන්නේ මැද ඉන්න **VLC Player** එක අර Chrome ගාවටම උඩට තල්ලු කරන එකයි.
    
- දැන් VLC එකට පල්ලෙහායින් අර විසිරී තිබුණු 50MB කෑලි දෙක එකතු වෙලා 100MB ලොකු තනි ඉඩක් හැදෙනවා.
    
- දැන් ඔයාගේ **Calculator (80MB)** එක කිසිම ප්‍රශ්නයක් නැතුව ඒ ඉඩට දාන්න පුළුවන්.



**Paging** කියන්නේ **External Fragmentation** කියන විශාල ගැටලුව සම්පූර්ණයෙන්ම විසඳන්න භාවිතා කරන ඉතාමත් දියුණු මතක කළමනාකරණ (**Memory Management**) ක්‍රමවේදයක්. අපි මේක සරලව තේරුම් ගනිමු.

---

### 1. External Fragmentation සහ එහි ගැටලුව

අපි කලින් කතා කළා වගේ, **External Fragmentation** කියන්නේ මතකයේ (RAM) අවශ්‍ය කරන සම්පූර්ණ ඉඩ ප්‍රමාණය තිබුණත්, එම ඉඩ එක දිගට (**Contiguous**) නොමැති වීම නිසා අලුත් වැඩසටහනක් මතකයට පටවා ගැනීමට නොහැකි වීමයි.

---

### 2. Paging යනු කුමක්ද?

**Paging** මගින් කරන්නේ වැඩසටහනක් (Process) මතකයේ එක දිගටම තිබිය යුතුයි කියන කොන්දේසිය ඉවත් කිරීමයි. මෙහිදී මතකය කොටස් දෙකකට බෙදනු ලබනවා:

- **Frames (Physical Memory):** සැබෑ **RAM** එක ස්ථාවර ප්‍රමාණයේ (**Fixed-sized**) කුඩා කොටස් වලට බෙදන අතර ඒවා **Frames** ලෙස හඳුන්වයි.
    
- **Pages (Logical Memory):** වැඩසටහනක් (Process) ද දකින ආකාරයට එම ප්‍රමාණයේම කුඩා කොටස් වලට බෙදන අතර ඒවා **Pages** ලෙස හඳුන්වයි.
    

---

### 3. Paging ක්‍රියාත්මක වන ආකාරය

1. **Equal Size:** සෑම විටම **Page** එකක ප්‍රමාණය සහ **Frame** එකක ප්‍රමාණය සමාන වේ.
    
2. **Non-contiguous Allocation:** වැඩසටහනක් මතකයට පටවන විට, එහි ඇති **Pages** එක දිගට තිබීම අත්‍යවශ්‍ය නැත. RAM එකේ හිස්ව පවතින ඕනෑම **Frame** එකකට ඕනෑම **Page** එකක් ඇතුළත් කළ හැකියි.
    
3. **Page Table:** වැඩසටහනක කුමන **Page** එක RAM එකේ කුමන **Frame** එකේද තියෙන්නේ කියලා සටහන් කරගන්නා "සිතියම" තමයි **Page Table** එක කියන්නේ.
    
4. **Translation:** CPU එක මගින් **Logical Address** එකක් ලබා දුන් විට, **Page Table** එක පාවිච්චි කරලා එය සැබෑ **Physical Address** එකක් බවට පත් කරනවා.
    

---

### ඇයි Paging නිසා External Fragmentation නැති වෙන්නේ?

හිතන්න ඔයාගේ **RAM** එකේ තැනින් තැන **1MB** කෑලි 10ක් තියෙනවා කියලා. දැන් ඔයාට **10MB** සොෆ්ට්වෙයාර් එකක් රන් කරන්න ඕනේ.

- **පරණ ක්‍රමයේදී:** එක දිගට 10MB නැති නිසා ඒක රන් කරන්න බැහැ.
    
- **Paging ක්‍රමයේදී:** අර 10MB සොෆ්ට්වෙයාර් එක 1MB බැගින් වූ **Pages** 10කට කඩනවා. දැන් අර විසිරී තිබුණු 1MB **Frames** 10 ඇතුළට මේ පේජස් 10 දාන්න පුළුවන්.
    

**සාරාංශය:** **Paging** නිසා මතකයේ ඉඩ අපතේ යන්නේ නැහැ, මොකද වැඩසටහනක් කෑලි වලට කඩලා (Pages) මතකයේ තියෙන ඕනෑම හිස් සිදුරකට (Frames) දාන්න පුළුවන් නිසයි.
