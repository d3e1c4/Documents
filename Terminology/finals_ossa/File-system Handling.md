### 1. File System එකක් යනු කුමක්ද?

**File System** එකක් කියන්නේ Operating System එකක තියෙන, දත්ත (Data) පාලනය කරන ප්‍රධාන පද්ධතියයි.

- එය දත්ත ගබඩා කරන්නේ කොහොමද (**Store**), පිළිවෙලකට සකසන්නේ කොහොමද (**Organize**) සහ අවශ්‍ය වෙලාවට නැවත ලබා ගන්නේ කොහොමද (**Retrieve**) කියන දේ තීරණය කරනවා.
    
- මෙය පරිශීලකයා (User) සහ Hardware (Hard Disk/SSD) අතර **Abstraction Layer** එකක් විදියට වැඩ කරනවා. ඒ කියන්නේ, දත්ත Hard Disk එකේ භෞතිකව (Physically) තැන්පත් වෙලා තියෙන සංකීර්ණ විදිය ගැන පරිශීලකයාට වද වෙන්න අවශ්‍ය නැහැ; පද්ධතිය එය සරලව අපිට පෙන්වනවා.
    

### 2. File එකක් යනු කුමක්ද? (File Concept)

- **Smallest Unit of Logical Storage:** Operating System එකක් තුළ දත්ත ගබඩා කිරීමට පාවිච්චි කරන කුඩාම "තාර්කික ඒකකය" තමයි **File** එකක් කියන්නේ.
    
- **Collection of Related Information:** එකකට එකක් සම්බන්ධ තොරතුරු සමූහයක් **Secondary Storage** (Hard Disk වැනි) මත ගබඩා කර ඇති විට එය File එකක් ලෙස හඳුන්වනවා.
    

### 3. File එකක් තුළ තිබිය හැකි දේවල්:

ඕනෑම File එකක් එය සාදන පුද්ගලයා (Creator) අනුව සහ එහි අරමුණ අනුව වෙනස් වෙනවා. ප්‍රධාන වශයෙන් මේවායේ තිබිය හැක්කේ:

- **Text:** ලිපි ලේඛන (Documents).
    
- **Numbers:** දත්ත ගොනු (Data files).
    
- **Binary Data:** පින්තූර (Images) සහ වීඩියෝ (Videos).
    
- **Executable Programs:** පරිගණකයේ ක්‍රියාත්මක කළ හැකි වැඩසටහන්.
    

### 4. OS එක File එකක් දකින ආකාරය:

අපි File එකක් ඇතුළේ දකින්නේ අකුරු හෝ රූප වුණත්, Operating System එක ඒ හැම දෙයක්ම දකින්නේ **Sequence of bytes** (බයිට් පෙළක්) හෝ **Records** එකතුවක් විදියටයි.

**සාරාංශය:** **File System** එක කියන්නේ දත්ත කළමනාකරණය කරන "ක්‍රමය" වන අතර, **File** එකක් කියන්නේ ඒ දත්ත අඩංගු වන "ගබඩාවයි".


**File Attributes** කියන්නේ සෑම **File** එකකටම අදාළව පවතින "අමතර තොරතුරු" (Metadata) වලටයි. සරලවම කිව්වොත්, මේක හරියට පුද්ගලයෙකුගේ හැඳුනුම්පතක් (ID Card) වගේ. එහි එම පුද්ගලයා ගැන විස්තර අඩංගු වෙනවා.

Operating System එක මගින් **Files** කාර්යක්ෂමව පාලනය කිරීමට මේවා පාවිච්චි කරනවා. ප්‍රධාන **Attributes** කිහිපය මෙන්න:

1. **Name:** අපිට (පරිශීලකයාට) කියවන්න සහ හඳුනාගන්න පුළුවන් නම (**Human-readable identifier**). උදා: `MyPhoto.jpg`.
    
2. **Identifier:** පද්ධතිය ඇතුළේ එම **File** එක අනෙක් ඒවායින් වෙන් කර හඳුනා ගැනීමට දෙන විශේෂ අංකයක් (**Unique internal ID**). පද්ධතිය ඇතුළේ වැඩ කරන්නේ මේ අංකයෙන්.
    
3. **Location:** **Storage device** එකේ (Hard Disk/SSD) මෙම **File** එක ඇත්තටම ගබඩා වෙලා තියෙන ස්ථානයට අදාළ තොරතුරු.
    
4. **Size:** **File** එකේ දැනට පවතින විශාලත්වය (**Current size**). මෙය සාමාන්‍යයෙන් Bytes, KB හෝ MB වලින් මනිනවා.
    
5. **Protection:** මෙම **File** එක පාවිච්චි කිරීමට, කියවීමට (Read) හෝ වෙනස් කිරීමට (Write) අවසර ඇත්තේ කාටද යන්න තීරණය කරන තොරතුරු (**Access permissions**).
    
6. **Time & Date:** **File** එක මුලින්ම හැදූ දිනය (Creation), අවසාන වරට වෙනස් කළ දිනය (Modification) වැනි තොරතුරු (**Timestamps**).
    

### මේවා ඇයි වැදගත් වෙන්නේ?

- **Security:** අනවශ්‍ය පුද්ගලයන් දත්ත බැලීම වැළැක්වීමට.
    
- **Tracking:** **File** එකක සිදුවන වෙනස්කම් ගැන සටහන් තබා ගැනීමට.
    
- **Organization:** **Files** පිළිවෙලකට ගොනු කර තබා ගැනීමට.

**Directory Structure** එක ගැන ඔයා දීලා තියෙන කරුණු සරලව මෙන්න මේ විදියට තේරුම් ගන්න පුළුවන්:

සරලවම කිව්වොත් **Directory** එකක් කියන්නේ අපි සාමාන්‍යයෙන් පාවිච්චි කරන **Folder** එකකටයි. හැබැයි Operating System එක ඇතුළේදී මෙය වෙනත් **Files** ගැන තොරතුරු ගබඩා කරලා තියාගන්න විශේෂිත **File** වර්ගයක් විදියට සලකනවා.

මේක හරියට පොතක තියෙන **Table of contents** (පටුන) එකක් වගේ. පොතේ තියෙන පාඩම් තියෙන්නේ කොයි පිටුවලද කියලා පටුනෙන් බලනවා වගේ, පද්ධතියේ තියෙන Files තියෙන්නේ කොහේද කියලා බලන්න **Directory** එක උදවු වෙනවා.

### Directories මගින් සිදුකරන ප්‍රධාන කාර්යයන් (Functions):

1. **Organize files logically:** **Files** අවුල් සහගතව තියෙන්න ඇරලා නැතුව, ඒවායේ ස්වභාවය අනුව පිළිවෙලකට සකස් කරලා තියාගන්න උදවු වෙනවා. (උදාහරණයක් විදියට, ඔයාගේ සින්දු ඔක්කොම "Songs" කියන **Directory** එකට දාගන්න පුළුවන්).
    
2. **Store file metadata:** අපි කලින් කතා කරපු **File metadata** (නම, size එක, අයිතිකරු වැනි දේවල්) ගබඩා වෙලා තියෙන්නේ මේ **Directory** එක ඇතුළෙයි.
    
3. **Enable efficient searching:** පරිගණකයේ තියෙන දහස් ගණනක් Files අතරින් අපිට අවශ්‍ය එක ඉක්මනින් සොයාගන්න (**Searching**) මේක උදවු වෙනවා. **Directory** එකක් නැත්නම් මුළු **Hard disk** එකම පීරන්න වෙනවා.
    

### මේකෙන් ලැබෙන වාසිය:

මේ **Directory Structure** එක නිසා **File management** එක ගොඩක් පහසු වෙනවා වගේම පද්ධතිය වඩාත් ක්‍රමවත් (**Structured**) වෙනවා.


### 1. File එකක් සහ Directory එකක් අතර සරල වෙනස

- **File:** මෙහි අඩංගු වෙන්නේ ඇත්තම දත්ත (**Actual Data**). උදාහරණයක් විදියට ඔයා ලියපු ලේඛනයක් (Text), ඔයාගේ පින්තූරයක් (Image) හෝ සින්දුවක් (Audio). මේවා පරිශීලකයාට ප්‍රයෝජනවත් තොරතුරු ගබඩා කරගන්නා ඒකකයයි.
    
- **Directory (Folder):** මෙය ඇත්තටම දත්ත ගබඩා කරන තැනකට වඩා, **Files** සමූහයක් පිළිවෙලකට තබාගන්නා "බහාලුමක්" (**Container**) වගේ දෙයක්.
    

### 2. ඇයි OS එක Directory එකක් "විශේෂිත File එකක්" විදියට සලකන්නේ?

පරිගණකයේ **Hard Disk** එකට හෝ **SSD** එකට "Folder" එකක් කියන්නේ මොකක්ද කියලා තේරෙන්නේ නැහැ. එයට පේන්නේ දත්ත ගබඩා කර ඇති **Storage Blocks** විතරයි. ඒ නිසා **Operating System** එක මේ වැඩේ පහසු කරගන්න උපක්‍රමයක් පාවිච්චි කරනවා:

- **Index එකක් වගේ:** හිතන්න පොතක තියෙන **Table of contents** (පටුන) ගැන. පටුන කියන්නෙත් පොතේම තියෙන එක පිටුවක් (File එකක්). හැබැයි ඒ පිටුවේ තියෙන්නේ කතාන්දර නෙවෙයි, "අහවල් පාඩම තියෙන්නේ අහවල් පිටුවේ" කියන තොරතුරු ටික විතරයි.
    
- **Directory එකක අන්තර්ගතය:** OS එක **Directory** එකක් නිර්මාණය කරන්නේ File එකක් විදියටමයි. හැබැයි ඒ File එක ඇතුළේ ලියවිලා තියෙන්නේ අපි දාන දත්ත නෙවෙයි. එහි තියෙන්නේ එම Directory එක ඇතුළේ තියෙන අනෙක් **Files වල නම්** සහ ඒවා Hard Disk එකේ **ගබඩා වෙලා තියෙන තැන් (Address/Pointer)** පිළිබඳ ලැයිස්තුවක් පමණයි.
    

### 3. සරල උදාහරණයක් (Analogy)

හිතන්න ඔයා ගාව ලොකු පෙට්ටියක් තියෙනවා කියලා.

- ඒ පෙට්ටිය ඇතුළේ තියෙන **ලියුම්** තමයි **Files**.
    
- ඔයා ඒ ලියුම් ටික වර්ග කරලා දාන **කවර (Envelope)** තමයි **Directories**.
    
- පරිගණකයට මේ කවරයත් එක වස්තුවක් (Object). හැබැයි ඒ කවරය ඇතුළේ තියෙන්නේ තවත් ලිපි ලේඛන නෙවෙයි, "මේ කවරය ඇතුළේ අහවල් ලියුම් ටික තියෙනවා" කියන **Metadata** ටික විතරයි. ඒ නිසයි ඒක **Special Type of File** එකක් වෙන්නේ.
    

**සාරාංශය:** **File** එකක් ඇතුළේ **User Data** (අපිට අවශ්‍ය දේ) තියෙනවා. **Directory** එකක් කියන විශේෂිත File එක ඇතුළේ තියෙන්නේ අනෙක් Files සොයාගන්න උදවු වෙන **System Data/Index** එකක්.

**File Operations** කියන්නේ **Operating System** එකක් මගින් **File** එකක් පාලනය කිරීමට හෝ වෙනස් කිරීමට අපිට ලබා දෙන මූලික අවස්ථා වලටයි. මේවා සරලව මෙහෙම තේරුම් ගනිමු:

1. **Create (නිර්මාණය කිරීම):** අලුතින් **File** එකක් සෑදීම. මෙහිදී OS එක **Storage** එකේ (Hard Disk/SSD) අවශ්‍ය ඉඩ වෙන් කර, ඒ සඳහා නමක් ලබා දෙනවා.
    
2. **Open (විවෘත කිරීම):** ඕනෑම **File** එකක් පාවිච්චි කිරීමට කලින් එය "විවෘත" කළ යුතුයි. මෙතැනදී OS එක එම **File** එකට අදාළ තොරතුරු මතකයට (**Memory**) ගෙන එය වැඩ කිරීමට සූදානම් කරනවා.
    
3. **Read (කියවීම):** **File** එකක් ඇතුළේ ගබඩා කරලා තියෙන දත්ත ලබා ගැනීම හෝ කියවීම මෙයින් සිදු කරනවා.
    
4. **Write (ලිවීම):** අලුතින් දත්ත එකතු කිරීම හෝ තිබෙන දත්ත වෙනස් කිරීම (**Modify**) මෙයට අයත් වෙනවා.
    
5. **Seek (සොයා යාම):** මේක වැදගත්. **File** එකක් ඇතුළේ අපි දැනට ඉන්න තැන පෙන්වන **File Pointer** එකක් තියෙනවා. **Seek** මගින් කරන්නේ ඒ **Pointer** එක අපිට අවශ්‍ය තැනකට (උදාහරණයක් විදියට File එකේ මැදට හෝ අගට) ගෙන යාමයි.
    
6. **Close (වසා දැමීම):** අපේ වැඩ අවසන් වූ පසු **File** එක වසා දැමිය යුතුයි. එවිට OS එක ඒ වෙනුවෙන් වෙන් කර තිබූ **Resources** (මතකය වැනි දෑ) නිදහස් කරනවා.
    
7. **Delete (මකා දැමීම):** අනවශ්‍ය **File** එකක් සම්පූර්ණයෙන්ම පද්ධතියෙන් ඉවත් කිරීම.
    

### මේවා ක්‍රියාත්මක වන්නේ කොහොමද?

මේ හැම **Operation** එකක්ම ක්‍රියාත්මක වෙන්නේ **System Calls** හරහායි. ඒ කියන්නේ අපේ වැඩසටහනක් (**Application**) මේ වැඩක් කරගන්න ඕන වුණාම **Operating System** එකෙන් කරන "ඉල්ලීමක්" තමයි **System Call** එකක් කියන්නේ.

**Open Files** කළමනාකරණය කියන්නේ Operating System එකක් දැනට විවෘත කරලා තියෙන **Files** ගැන විස්තර ඉතාමත් ක්‍රමවත්ව තබා ගන්නා ආකාරයයි.

පරිගණකයේ වැඩසටහනක් (**Process**) මගින් **File** එකක් **Open** කළාම, OS එක ඒ **File** එකේ තොරතුරු **Open-file table** කියන විශේෂ ලැයිස්තුවට ඇතුළත් කරගන්නවා. මේ වැඩේ කරන්නේ ඇයි සහ ඒකට මොන වගේ දත්තද ඕන වෙන්නේ කියලා අපි බලමු:

### 1. Open-file table එකේ තියෙන ප්‍රධාන දත්ත (Managed Data)

- **File Pointer:** මේකෙන් කරන්නේ අපි **File** එකේ අන්තිමට කියවපු (**Read**) හෝ ලියපු (**Write**) ස්ථානය සලකුණු කර තබා ගැනීමයි. එකම **File** එක **processes** දෙකකින් **open** කරලා තිබුණොත්, ඒ දෙකටම වෙන වෙනම **File Pointers** තියෙනවා.
    
- **File-open Count:** එකම **File** එක **processes** කීයක් පාවිච්චි කරනවද කියලා මේකෙන් ගණන් ගන්නවා. **Count** එක බින්දුව (0) වුණාම, ඒ කියන්නේ අන්තිම **process** එකත් **File** එක **Close** කළාම තමයි OS එක ඒ **File** එකේ දත්ත **Open-file table** එකෙන් අයින් කරන්නේ.
    
- **Disk Location:** **File** එක **Hard Disk** එකේ ඇත්තටම තියෙන්නේ කොහේද කියන තොරතුර මෙතැන ගබඩා වෙනවා. හැම වෙලේම **Disk** එක පීරන්න යන්නේ නැතුව ඉක්මනින් දත්ත ලබා ගන්න (**Cache**) මේක උදවු වෙනවා.
    
- **Access Rights:** අදාළ **process** එකට ඒ **File** එක කියවන්න (**Read**) විතරද අවසර තියෙන්නේ, නැත්නම් ලියන්නත් (**Write**) අවසර තියෙන්නේ කියලා මෙතැනදී පරීක්ෂා කරනවා.
    

### 2. මේ ක්‍රමයේ වාසි (Benefits)

    - විශාල දත්ත පද්ධති (**Databases**) සඳහා අනිවාර්යයෙන්ම පාවිච්චි වෙනවා.
        
- **උදාහරණ:**
    
    - **Database:** දහස් ගණනක් දත්ත අතරින් 100 වෙනි දත්තය (Record number 100) කෙලින්ම ලබා ගැනීම.
        
    - **CD / Spotify:** සින්දු ලැයිස්තුවක ඕනෑම සින්දුවක් උඩ Click කරලා ඒ සින්දුව කෙලින්ම අහන්න පුළුවන් වීම.

### සාරාංශය (Comparison)

|**ලක්ෂණය**|**Sequential Access**|**Direct Access**|
|---|---|---|
|**පිළිවෙල**|මුල සිට අගට අනුපිළිවෙලින්|ඕනෑම තැනකට කෙලින්ම|
|**වේගය**|මන්දගාමී (මැද දත්තයක් සෙවීමේදී)|ඉතා වේගවත්|
|**පහසුව**|සරලයි|සංකීර්ණයි (Addressing අවශ්‍ය නිසා)|

සාමාන්‍යයෙන් අපේ අවශ්‍යතාවය අනුව OS එක මේ ක්‍රම දෙකෙන් එකක් තෝරා ගන්නවා. උදාහරණයක් විදියට ඔයා ලොකු **Database** එකක් පාවිච්චි කරනවා නම් **Direct Access** භාවිතා කිරීමෙන් ඉතා වේගවත් ප්‍රතිඵල ලබා ගන්න පුළුවන්.

අපි කලින් කතා කළානේ **Directory** එකක් කියන්නේ "පටුනක්" (Index) වගේ වැඩ කරන විශේෂිත **File** වර්ගයක් කියලා. දැන් අපි බලමු මේ පටුන පද්ධතිය ඇතුළේ පිළිවෙලකට සකස් කරලා තියෙන විවිධ ක්‍රම (**Types of Directory Structures**) මොනවාද කියලා.

මේවා අපේ **File Operations** (Search, Create, Delete) වල වේගයට සහ පහසුවට සෘජුවම බලපානවා.

### 1. Single-level Directory (තනි මට්ටමේ පද්ධතිය)

- **පැහැදිලි කිරීම:** මෙතැනදී මුළු පද්ධතියටම තියෙන්නේ එකම එක **Directory** එකක් විතරයි. හැම පරිශීලකයෙකුගේම (User) හැම **File** එකක්ම තියෙන්නේ මේ එකම ලැයිස්තුවේ.
    
- **කලින් කරුණු සමඟ සම්බන්ධය:** මෙහිදී **Search** කරන එක හරිම මන්දගාමීයි, මොකද දහස් ගණන් Files අතරින් එකක් හොයන්න මුළු ලැයිස්තුවම බලන්න ඕන. ඒ වගේම එකම නම තියෙන Files දෙකක් (**Naming problem**) තියෙන්න බැහැ.
    

### 2. Two-level Directory (මට්ටම් දෙකේ පද්ධතිය)

- **පැහැදිලි කිරීම:** මෙහිදී සෑම පරිශීලකයෙකුටම (User) වෙන වෙනම **User File Directory (UFD)** එකක් තියෙනවා. ඒ හැම එකක්ම පාලනය කරන ප්‍රධාන **Master File Directory (MFD)** එකක් තියෙනවා.
    
- **කලින් කරුණු සමඟ සම්බන්ධය:** දැන් **Search** කිරීමේදී OS එක මුලින්ම අදාළ පරිශීලකයාගේ Directory එකට යනවා. ඒ නිසා වේගය වැඩියි. පරිශීලකයන් දෙන්නෙකුට එකම නමින් Files තියාගන්න පුළුවන් (Naming problem එක විසඳෙනවා).
    

### 3. Tree-structured Directory (ගසක හැඩයට විහිදුණු පද්ධතිය)

- **පැහැදිලි කිරීම:** මේක තමයි අද අපි Windows හෝ Linux වල දකින සාමාන්‍ය ක්‍රමය. මෙතැනදී **Directories** ඇතුළේ තවත් **Sub-directories** (Folders) තියෙන්න පුළුවන්. මෙය ගසක අතු විහිදෙනවා වගේ **Root** එකේ ඉඳන් පහළට විහිදෙනවා.
    
- **කලින් කරුණු සමඟ සම්බන්ධය:** අපි කලින් කතා කරපු **Traverse** (පද්ධතිය පුරා සැරිසැරීම) මෙතැනදී හරිම වැදගත්. අපිට **Path** එක (උදා: `C:\Users\Documents\file.txt`) භාවිතා කරලා නිවැරදි තැනට යන්න පුළුවන්. දත්ත පිළිවෙලකට ගොනු කරන්න (**Organize**) තියෙන හොඳම ක්‍රමය මෙයයි.
    

### 4. Graph-based Directory (ග්‍රාෆ් එකක් ලෙස සම්බන්ධ වුණු පද්ධතිය)

- **පැහැදිලි කිරීම:** මෙය **Tree structure** එකට වඩා නම්‍යශීලීයි. මෙහිදී එකම **File** එකකට හෝ **Directory** එකකට පද්ධතියේ ස්ථාන කිහිපයක ඉඳන් සම්බන්ධ වෙන්න පුළුවන්. (උදාහරණයක් විදියට **Shortcut** හෝ **Link** එකක් පාවිච්චි කරනවා වගේ).
    
- **කලින් කරුණු සමඟ සම්බන්ධය:** මෙහිදී **Acyclic Graph** (එකම තැන වටේ කැරකෙන්නේ නැති) ක්‍රමය පාවිච්චි කරනවා. මෙතැනදී **Delete** operation එකේදී ප්‍රවේශම් වෙන්න ඕන, මොකද එක තැනකින් File එකක් මැකුවොත් ඒකට සම්බන්ධ වෙලා තිබුණ අනෙක් "පාරවල්" (Links) වලට මොකද වෙන්නේ කියලා OS එක තීරණය කරන්න ඕන.
    

### සාරාංශය:

- **Single-level:** සරලයි, හැබැයි ලොකු පද්ධති වලට ගැලපෙන්නේ නැහැ.
    
- **Two-level:** පරිශීලකයන් කිහිප දෙනෙක් ඉන්න කොට හොඳයි.
    
- **Tree:** දත්ත පිළිවෙලකට බෙදා වෙන් කරලා තියන්න (Hierarchy) ඉතාම හොඳයි.
    
- **Graph:** ගොනු කිහිප දෙනෙක් අතර බෙදා හදා ගන්න (**Sharing**) පහසුයි.

**File Protection** කියන්නේ Operating System එකක තියෙන දත්ත සහ ගොනු (Files) අනවශ්‍ය පුද්ගලයන්ගෙන් ආරක්ෂා කරගන්නා ආකාරයයි. මෙහිදී සිදුවන්නේ අදාළ ගොනුව භාවිතා කිරීමට අවසර ඇත්තේ කාටද (**Authorized users**) සහ ඔවුන්ට කළ හැක්කේ කුමක්ද යන්න තීරණය කිරීමයි.

අපි මෙය උදාහරණ සහිතව සරලව තේරුම් ගනිමු:

### 1. පිවිසුම් වර්ග (Types of Access)

සාමාන්‍යයෙන් Operating System එකක් මගින් File එකකට පිවිසිය හැකි ප්‍රධාන ආකාර 3ක් ලබා දෙනවා:

- **Read (කියවීම):** මෙම අවසරය ඇති පරිශීලකයෙකුට ගොනුව ඇතුළේ ඇති දේ බැලීමට පමණක් හැකියාව ලැබේ. එහි ඇති දත්ත වෙනස් කිරීමට හෝ මැකීමට නොහැක.
    
    - **උදාහරණ:** ඔයාට අන්තර්ජාලයෙන් ලැබුණු **Read-only PDF** එකක් කියවීම.
        
- **Write (ලිවීම):** මෙහිදී ගොනුව ඇතුළත ඇති දත්ත වෙනස් කිරීමට (**Modify**), අලුතින් ලිවීමට හෝ ගොනුව සම්පූර්ණයෙන් මැකීමට අවසර ලැබේ.
    
    - **උදාහරණ:** ඔයා Notepad එකක ලියපු **Assignment.txt** එකට අලුත් පේළියක් එකතු කර එය Save කිරීම.
        
- **Execute (ක්‍රියාත්මක කිරීම):** මෙම අවසරය ලැබෙන්නේ වැඩසටහන් (**Programs**) සඳහායි. ගොනුවක් වැඩසටහනක් ලෙස පද්ධතිය තුළ ක්‍රියාත්මක කිරීමට මෙයින් ඉඩ ලැබේ.
    
    - **උදාහරණ:** ඔයාගේ පරිගණකයේ ඇති ක්‍රීඩාවක් (Game) හෝ Chrome වැනි **Application** එකක් ඕපන් කිරීම.
        

---

### 2. Multi-user Systems වලදී මෙහි වැදගත්කම

අද කාලයේ බොහෝ පරිගණක පද්ධති වල පරිශීලකයන් කිහිප දෙනෙකු (Multi-user) සිටිනවා. එවැනි අවස්ථාවකදී **Protection mechanisms** ඉතාමත් තීරණාත්මකයි.

- **උදාහරණ:** හිතන්න ඔයා සහ ඔයාගේ සහෝදරයා එකම පරිගණකයේ ගිණුම් (**User Accounts**) දෙකක් පාවිච්චි කරනවා කියලා.
    
    - ඔයාගේ ගිණුමේ තියෙන පුද්ගලික පින්තූර (**Personal Photos**) ඔයාගේ සහෝදරයාගේ ගිණුමෙන් ලොග් වුණාම බලන්න බැරි වෙන්න OS එක වගබලා ගන්නවා.
        
    - ඒ වගේම, පද්ධතියේ ප්‍රධාන **System Files** වෙනත් සාමාන්‍ය පරිශීලකයන්ට මකා දැමිය නොහැකි ලෙස ආරක්ෂා කර තබනවා.
        

---

### 3. ප්‍රායෝගික ක්‍රමවේදය (Access Matrix/Control List)

OS එක මේ වැඩේ කරන්නේ ලැයිස්තුවක් තබා ගැනීමෙනුයි. ලිනක්ස් (Linux) වැනි පද්ධති වල මෙය `rwx` (Read, Write, Execute) ලෙස හඳුන්වනවා.

- **Owner (අයිතිකරු):** Read, Write, Execute යන අවසර තුනම තිබිය හැක.
    
- **Group (කණ්ඩායම):** Read සහ Execute පමණක් තිබිය හැක.
    
- **Others (අන් අය):** Read පමණක් හෝ කිසිදු අවසරයක් නොතිබිය හැක.
    

**සාරාංශය:** **File Protection** හරහා කරන්නේ දත්ත වල රහස්‍යභාවය (**Confidentiality**) සහ ආරක්ෂාව රැක ගැනීමයි.


Unix/Linux වල **Access Lists** සහ **Groups** පාවිච්චි කරලා ගොනු (Files) වල ආරක්ෂාව පාලනය කරන ආකාරය ඔයා දීපු කරුණු අනුව සරලව මෙන්න මෙහෙමයි:

Unix පද්ධතියක දත්ත ආරක්ෂා කිරීමට ප්‍රධාන මට්ටම් තුනක් සහ ක්‍රම තුනක් තියෙනවා.

### 1. Mode of Access (පිවිසුම් ආකාර)

ඕනෑම File එකක් සඳහා ලබා දිය හැකි අවසරයන් (**Permissions**) වර්ග 3කි:

- **Read (R):** File එක ඇතුළේ තියෙන දේ බැලීමට.
    
- **Write (W):** File එකේ දත්ත වෙනස් කිරීමට.
    
- **Execute (X):** File එක වැඩසටහනක් ලෙස ක්‍රියාත්මක (Run) කිරීමට.
    

---

### 2. Classes of Users (පරිශීලක පන්ති)

Unix වලදී මේ අවසරයන් ලබා දෙන්නේ පිරිස් තුනකට බෙදලයි:

- **Owner:** File එක නිර්මාණය කළ පුද්ගලයා.
    
- **Group:** එක්තරා නිශ්චිත කණ්ඩායමකට අයත් පිරිසක්.
    
- **Public (Others):** පද්ධතියේ ඉන්න අනිත් ඕනෑම කෙනෙක්.
    

---

### 3. Binary සහ Numerical Notation (අංකනය කරන ක්‍රමය)

විභාග වලදී මේ අවසරයන් පෙන්වන්න **Binary** (0 සහ 1) සහ ඒකෙන් ලැබෙන එකතුව (**Octal number**) පාවිච්චි කරනවා.

ඔයාගේ උදාහරණය බලමු:

- **Owner Access 7 (1 1 1):** මෙහි `1 1 1` කියන්නේ **RWX** තුනටම අවසර තියෙනවා කියන එකයි. (අගය = 4+2+1 = **7**)
    
- **Group Access 6 (1 1 0):** මෙහි `1 1 0` කියන්නේ **R** සහ **W** තියෙනවා, හැබැයි **X** (Execute) අවසරය නැහැ කියන එකයි. (අගය = 4+2+0 = **6**)
    
- **Public Access 1 (0 0 1):** මෙහි `0 0 1` කියන්නේ **R** සහ **W** නැහැ, පුළුවන් **X** (Execute) කරන්න විතරයි. (අගය = 0+0+1 = **1**)
    

---

### 4. ප්‍රායෝගිකව ක්‍රියාත්මක කිරීම (Practical Steps)

1. **Group එකක් සෑදීම:** මුලින්ම පද්ධති පරිපාලකයාට (**Manager/Admin**) කියලා විශේෂ නමක් තියෙන **Group** එකක් (උදා: `G`) සාදවා ගෙන, ඒකට අවශ්‍ය පරිශීලකයන්ව (**Users**) ඇතුළත් කරගන්නවා.
    
2. **Access එක තීරණය කිරීම:** ඔයා හදපු `game` කියන file එකට Owner ට, Group එකට සහ අනිත් අයට ලැබිය යුතු අවසරයන් මොනවාද කියලා තීරණය කරනවා.
    
3. **Group එක සම්බන්ධ කිරීම (`chgrp` විධානය):** ඔයා හදපු file එක අදාළ කණ්ඩායමට අයිති කරන්න මෙන්න මේ විධානය පාවිච්චි කරනවා:
    
    Bash
    
    ```
    chgrp G game
    ```
    
    මෙහි **`chgrp`** කියන්නේ **Change Group** යන්නයි. මේකෙන් කරන්නේ `game` කියන file එක `G` කියන group එකට සම්බන්ධ කිරීමයි (**Attach**).


**File-System Structure** එක මට්ටම් කිහිපයකට (Layers) බෙදා තිබීම නිසා Operating System එකකට ඉතාමත් සංකීර්ණ Hardware පාලනය කිරීම පහසු වෙනවා. මෙය හරියට ආයතනයක වැඩ කොටස් කිහිපයකට බෙදා ඇති ආකාරය වගේ දෙයක්.

අපි මේ ලේයර් 4 සරලව තේරුම් ගනිමු:

### 1. Application Layer (යෙදුම් මට්ටම)

- **පැහැදිලි කිරීම:** මේක තමයි පරිශීලකයා (User) ඉන්න තැන. අපි පාවිච්චි කරන VS Code, MS Word වැනි **Applications** මෙයට අයිතියි.
    
- **කාර්යය:** මෙහිදී අපි කරන්නේ "File එකක් Save කරන්න" හෝ "Open කරන්න" කියලා Operating System එකට **System Call** එකක් හරහා ඉල්ලීමක් කරන එකයි. අපිට පේන්නේ File එකේ නම විතරයි.
    

### 2. Logical File System (තාර්කික මට්ටම)

- **පැහැදිලි කිරීම:** මේක තමයි පද්ධතියේ "කළමනාකරු" වගේ වැඩ කරන්නේ.
    
- **කාර්යය:** මෙයා තමයි File එකක තියෙන **Metadata** (නම, අයිතිකරු, දිනය) සහ අපි කලින් කතා කරපු **Directory Structure** එක පාලනය කරන්නේ. File එකකට පිවිසීමට අවසර තියෙනවද (**Protection**) කියලා බලන්නෙත් මේ ලේයර් එකයි.
    

### 3. Physical File System / File Organization Module (භෞතික සංවිධාන මට්ටම)

- **පැහැදිලි කිරීම:** මෙයා තමයි File එකක් සහ එය Hard Disk එකේ ගබඩා වන ආකාරය අතර සම්බන්ධය පවත්වන්නේ.
    
- **කාර්යය:** File එකක් කියන්නේ බයිට් පෙළක්නේ. මේ ලේයර් එක මගින් ඒ බයිට් ටික Hard Disk එකේ තියෙන **Logical blocks** (උදා: block 10, block 11) බවට පත් කරනවා. File එකක තියෙන දත්ත කෑලි Hard Disk එකේ කොයි කොයි තැන්වලද තියෙන්නේ කියලා දන්නේ මෙයා.
    

### 4. Device Drivers / I/O Control (උපාංග පාලන මට්ටම)

- **පැහැදිලි කිරීම:** මේක තමයි Hardware එකටම සම්බන්ධ ලේයර් එක.
    
- **කාර්යය:** ඉහළ ලේයර් වලින් එන උපදෙස් (උදා: "block 10 කියවන්න") Hard Disk එකට තේරෙන ඉලෙක්ට්‍රොනික සංඥා හෝ විධානයන් (**Read/Write Commands**) බවට පත් කරන්නේ මෙයා. එක් එක් Hardware වර්ගය (SSD, HDD, USB) අනුව මේ **Device Drivers** වෙනස් වෙනවා.
    

---

### ඇයි මේ Layered Design එක මෙච්චර වැදගත්? (Benefits)

1. **Modularity (මොඩියුලර් ස්වභාවය):** ඔයා ඔයාගේ පරණ Hard Disk එක අයින් කරලා අලුත් SSD එකක් දැම්මොත්, ඔයාට වෙනස් කරන්න වෙන්නේ **Device Driver** ලේයර් එක විතරයි. උඩ තියෙන **Logical File System** එක හෝ **Application Layer** එක වෙනස් කරන්න අවශ්‍ය වෙන්නේ නැහැ.
    
2. **Efficiency (කාර්යක්ෂමතාව):** සෑම ලේයර් එකකටම පැහැදිලිව වෙන් කරපු වැඩ කොටසක් තියෙන නිසා දෝෂ හඳුනා ගැනීම සහ දත්ත හැසිරවීම ඉතාමත් වේගවත් වෙනවා.
    
3. **Abstraction:** පරිශීලකයාට (User) Hard Disk එක ඇතුළේ block වලට වැඩ සිද්ධ වෙන සංකීර්ණ විදිය ගැන වද වෙන්න අවශ්‍ය නැහැ; ඒ දේවල් යට තියෙන ලේයර් වලින් බලාගන්නවා.

**File Control Block (FCB)** කියන්නේ Operating System එකක් මගින් සෑම **File** එකක් සඳහාම පවත්වාගෙන යනු ලබන ඉතාමත් වැදගත් දත්ත සමූහයකි. සරලවම කිව්වොත්, මෙය සෑම File එකකටම අදාළව ඇති "හැඳුනුම්පත" හෝ "තොරතුරු වාර්තාව" වැනිය.

පරිගණකයේ ගොනු දහස් ගණනක් තිබිය හැකි නිසා, ඒ සෑම එකක්ම පාලනය කිරීමට OS එකට මේ **FCB** එක උදවු වෙනවා. අපි ඔයා දීපු වගුවේ තියෙන කරුණු එකින් එක බලමු:

### 1. FCB එකක අඩංගු ප්‍රධාන තොරතුරු

- **File Permissions:** අපි කලින් කතා කරපු **Read, Write, Execute** අවසරයන් මෙහි සටහන් වේ.
    
- **File Dates (Timestamps):** ගොනුව මුලින්ම සෑදූ දිනය (**Create**), අවසාන වරට එය විවෘත කළ දිනය (**Access**) සහ අවසාන වරට වෙනස් කළ දිනය (**Write**) මෙහි අඩංගු වේ.
    
- **File Owner, Group, ACL:** මේ File එක අයිති පරිශීලකයා (**Owner**), කණ්ඩායම (**Group**) සහ එම File එකට පිවිසිය හැකි අයගේ සම්පූර්ණ ලැයිස්තුව (**Access Control List - ACL**) මෙහි ඇත.
    
- **File Size:** File එකේ දැනට පවතින මුළු විශාලත්වය (Bytes වලින්).
    
- **File Data Blocks or Pointers:** මෙය තමයි වැදගත්ම කොටස. File එකක ඇත්තම දත්ත (**Actual Data**) Hard Disk එකේ කොයි කොයි තැන්වල (Blocks) ද ගබඩා වෙලා තියෙන්නේ කියන එක මෙහි ඇති **Pointers** (දර්ශක) හරහා OS එක දැනගන්නවා.
    

---

### 2. FCB එකක ඇති වැදගත්කම (Why it's important?)

- **Metadata Storage:** File එක ඇතුළේ තියෙන ඇත්තම දත්ත වලට අමතරව එම File එක ගැන තියෙන තොරතුරු (**Metadata**) ගබඩා කරන්නේ මෙතැනයි.
    
- **Efficient Handling:** OS එකට සෑම වෙලාවකම File එක ඇතුළේ තියෙන හැම දත්තයක්ම කියවන්න අවශ්‍ය වෙන්නේ නැහැ. **FCB** එක බැලීමෙන් පමණක් File එකේ නම, විශාලත්වය සහ අවසරයන් ඉක්මනින් දැනගන්න පුළුවන්.
    
- **Location Identification:** File එකක් Open කළ විට, එහි දත්ත Hard Disk එකේ ඇති නිවැරදි ස්ථානයෙන් ලබා ගැනීමට මේ **Data Block Pointers** උදවු වෙනවා.
    

### 3. සරල උදාහරණයක් (Analogy)

හිතන්න ඔයා ගාව ලොකු **බඩු පෙට්ටියක්** තියෙනවා කියලා. ඒ පෙට්ටිය ඇතුළේ තියෙන බඩු තමයි **File Data**. පෙට්ටියෙන් පිටත අලවා ඇති **Label** එක තමයි **FCB** එක. ඒ Label එකේ පෙට්ටියේ අයිතිකරු, බර (Size), සහ ඇතුළේ තියෙන්නේ මොනවාද කියලා ලියලා තියෙනවා. පෙට්ටිය ඇරලා බලන්නේ නැතුව ඒ Label එක බැලීමෙන් පෙට්ටිය ගැන විස්තර දැනගන්න පුළුවන් වගේ තමයි OS එක **FCB** එක පාවිච්චි කරන්නෙත්.

---

### විභාගය සඳහා විශේෂ කරුණක්:

Linux/Unix වැනි පද්ධති වල මේ **FCB** එක හඳුන්වන්නේ **inode** (index node) කියන නමිනුයි. ඔයාගේ සටහනේ "Typically, inode number" කියලා තියෙන්නේ ඒකයි.


**File Allocation Method** කියන්නේ Operating System එකක් මගින් Hard Disk එකේ ඇති ඉඩ (**Disk Blocks**), ගොනු (Files) සඳහා වෙන් කර දෙන ආකාරයයි.

Hard Disk එකක් කියන්නේ කුඩා කොටස් (**Blocks**) රාශියක එකතුවක්. අපිට File එකක් Save කරන්න ඕන වුණාම, OS එක තීරණය කරන්න ඕන ඒ File එකේ දත්ත මේ කුඩා Blocks වල ගබඩා කරන්නේ කොහොමද කියලා. මේ සඳහා ප්‍රධාන ක්‍රම 3ක් තියෙනවා:

---

### 1. Contiguous Allocation (සන්තතිකව වෙන් කිරීම)

මෙහිදී File එකකට අවශ්‍ය Blocks සියල්ලම Hard Disk එකේ එක පෙළට (පිට පිට) පිහිටි ස්ථාන වලින් ලබා දෙයි.

- **වැඩ කරන හැටි:** ඔයාට Block 3ක විශාලත්වයක් තියෙන File එකක් Save කරන්න ඕන නම්, OS එක විසින් එක ළඟ තියෙන හිස් Block 3ක් (උදා: 10, 11, 12) සොයාගෙන එය ලබා දෙයි.
    
- **වාසි:** ඉතාමත් වේගවත්. විශේෂයෙන් **Direct Access** සඳහා ඉතා සුදුසුයි.
    
- **අවාසි:** **External Fragmentation** ඇති වේ. ඒ කියන්නේ අතරමැද හිස් ඉඩ තිබුණත්, ඒවා එක පෙළට නැතිනම් ලොකු File එකක් Save කරන්න බැරි වීම. එමෙන්ම File එකක විශාලත්වය පසුව වැඩි කිරීමට අපහසුය.
    
- **උදාහරණ:** පැරණි CD/DVD වල දත්ත ගබඩා වන ආකාරය.

### 2. Linked Allocation (සම්බන්ධිතව වෙන් කිරීම)

මෙහිදී File එකක දත්ත Blocks වල විසිරී තිබිය හැක. සෑම Block එකක්ම ඊළඟ Block එක තියෙන තැන පෙන්වන **Pointer** එකක් (දර්ශකයක්) තබා ගනී.

- **වැඩ කරන හැටි:** මෙය හරියට "සැඟවුණු අරමුදල් සොයන" සෙල්ලමක් වගෙයි. පළමු Block එකේ දත්ත තියෙනවා, ඒකෙ අග තියෙන ලිපිනයෙන් කියනවා ඊළඟ කොටස තියෙන්නේ කොහේද කියලා.
    
- **වාසි:** **Fragmentation** ගැටලුවක් නැහැ. ඕනෑම තැනක තියෙන හිස් Block එකක් පාවිච්චි කරන්න පුළුවන්. File එකක Size එක ඕන වෙලාවක වැඩි කරන්න ලේසියි.
    
- **අවාසි:** පාවිච්චි කළ හැක්කේ **Sequential Access** විතරයි. මැද තියෙන දත්තයක් බලන්න නම් මුල ඉඳන්ම පීරන්න ඕනේ. එක Pointer එකක් හරි නැති වුණොත් මුළු File එකම පාවිච්චි කරන්න බැරි වෙනවා.
    
- **උදාහරණ:** සරල ලැයිස්තු (Linked Lists) ආකාරයට දත්ත ගබඩා කිරීම.

### 3. File Allocation Table (FAT)

මෙය **Linked Allocation** වලම දියුණු කළ ආකාරයකි. මෙහිදී සෑම Block එකකම Pointer එක තබා ගන්නවා වෙනුවට, සියලුම Blocks වල ලිපින අඩංගු වෙනම වගුවක් (**Table**) Hard Disk එකේ මුල පවත්වාගෙන යයි.

- **වැඩ කරන හැටි:** OS එක මුලින්ම මේ **FAT table** එක කියවනවා. එහි File එකට අදාළ Blocks මොනවාද සහ ඒවා පිහිටි අනුපිළිවෙල සඳහන් වෙනවා.
    
- **වාසි:** Linked allocation වලට වඩා **Direct Access** (තැනින් තැනට පැනීම) වේගවත් වේ. මොකද OS එක මුළු Disk එකම පීරන්නේ නැතුව අර Table එක විතරක් බලන නිසයි.
    
- **අවාසි:** මේ Table එක ගබඩා කිරීමට Disk එකේ වෙනම ඉඩක් වෙන් කරන්න සිදු වේ. Table එකට හානි වුණොත් මුළු Disk එකේම දත්ත වලට ගැටලු ඇති වේ.
    
- **උදාහරණ:** අපි පාවිච්චි කරන USB Flash Drives (පෙන් ඩ්‍රයිව්) සහ SD Cards වල තවමත් මේ FAT32 ක්‍රමය බහුලව පාවිච්චි වෙනවා.

### සාරාංශය (Comparison)

| **ලක්ෂණය**  | **Contiguous**      | **Linked**      | **FAT**                          |
| ----------- | ------------------- | --------------- | -------------------------------- |
| **පිළිවෙල** | එක දිගට             | විසිරී ඇත       | විසිරී ඇත (නමුත් ලැයිස්තුවක් ඇත) |
| **Access**  | Sequential & Direct | Sequential Only | Sequential & Direct              |
| **වේගය**    | ඉතා වේගවත්          | මන්දගාමී        | මධ්‍යස්ථ                         |
| **භාවිතය**  | CD / DVD            | පරණ පද්ධති      | USB / SD Cards                   |

**Contiguous Allocation Method** කියන්නේ ගොනු (Files) Hard Disk එකේ ගබඩා කරන සරලම සහ වේගවත්ම ක්‍රමයයි. මෙහිදී සිදුවන්නේ File එකකට අවශ්‍ය සියලුම **Blocks** ටික Hard Disk එකේ එක පෙළට (එක ළඟ) පිහිටි ස්ථාන වලින් වෙන් කර දීමයි.

අපි මේකේ වැදගත් කරුණු ටික කොටස් වශයෙන් තේරුම් ගනිමු:

### 1. වැඩ කරන ආකාරය (How it works)

මෙම ක්‍රමයේදී OS එකට මතක තබා ගත යුත්තේ කරුණු 2ක් පමණයි:

1. **Starting Location:** File එක පටන් ගන්නා Block එකේ අංකය.
    
2. **Length:** File එකට Blocks කීයක් අයිතිද යන්න.
    

- **උදාහරණයක්:** File එකක් 50 වෙනි Block එකෙන් පටන් ගන්නවා නම් සහ එහි දිග (Length) 5ක් නම්, එය 50, 51, 52, 53, 54 යන Blocks වල එක පෙළට ගබඩා වේ.

### 2. වේගය සහ I/O Operations (Performance)

මේ ක්‍රමයේ තියෙන ලොකුම වාසිය තමයි වේගය. අපිට අවශ්‍ය ඕනෑම තැනකට කෙලින්ම (**Direct Access**) යන්න පුළුවන්.

- **8 වෙනි Block එකට යාමට අවශ්‍ය I/O ගණන:** මෙහිදී OS එක දන්නවා File එක පටන් ගන්නේ කොහෙන්ද කියලා. ඒ නිසා 8 වෙනි එක හොයන්න pointers පස්සේ යන්න ඕනේ නැහැ. සූත්‍රයක් මගින් ලිපිනය කෙලින්ම ගණනය කරලා (**Address Calculation**), එකම එක **I/O operation** එකකින් ඒ තැනට පනින්න පුළුවන්.
    
- **මුළු File එකම (Blocks 12ක්) කියවීමට:** Blocks 12ම පිළිවෙලින් කියවිය යුතු නිසා **12 I/O operations** අවශ්‍ය වේ.

### 3. මෙහි ඇති ගැටලු (Problems)

හැබැයි මේ ක්‍රමයේ අවාසි කිහිපයක් තියෙනවා:

- **External Fragmentation:** අතරමැද හිස් ඉඩ (Free blocks) විසිරී තිබුණත්, ඒවා එක පෙළට නැතිනම් ලොකු File එකක් Save කරන්න බැරි වෙනවා.
    
- **Knowing File Size:** File එකක් හදනකොටම ඒක ඉස්සරහට කොච්චර ලොකු වෙනවද කියලා අපි දන්නේ නැහැ. එක පෙළට ඉඩ වෙන් කළාට පස්සේ File එක තව ලොකු කරන්න වුණොත්, ඊළඟට තියෙන Block එක දැනටමත් වෙනත් File එකක් පාවිච්චි කරනවා වෙන්න පුළුවන්.
    
- **Compaction:** මේ විසිරුණු හිස් ඉඩවල් එකතු කරලා ලොකු ඉඩක් හදන්න නම්, තැටියේ තියෙන Files එහා මෙහා කරන්න වෙනවා. මේකට **Compaction** කියලා කියනවා. ඒ වෙලාවට පද්ධතිය ක්‍රියා විරහිත කරන්න (**Downtime**) සිදුවිය හැකියි.

### 4. Extent-Based Systems (දියුණු කළ ක්‍රමය)

නවීන පද්ධති මේ ගැටලු විසඳන්න **Extents** කියන ක්‍රමය පාවිච්චි කරනවා.

- **Extent යනු:** එක පෙළට තියෙන Blocks සමූහයකි.
    
- **වැඩ කරන හැටි:** මුලින්ම File එකකට ලොකු **Extent** එකක් (එක පෙළට තියෙන Blocks කෑල්ලක්) දෙනවා. ඒක පිරුණොත්, තැටියේ වෙනත් තැනක තියෙන තවත් එක පෙළට තියෙන Blocks කෑල්ලක් (**Another Extent**) ලබා දෙනවා.
    
- **වාසිය:** මෙතැනදී මුළු File එකම එක දිගට තියෙන්න ඕනේ නැහැ. හැබැයි File එක ඇතුළේ තියෙන දත්ත "කෑලි" (Extents) එක දිගට පවතින නිසා වේගයත් රැකෙනවා, **Fragmentation** ගැටලුවත් පාලනය වෙනවා.
    

**සාරාංශය:** **Contiguous Allocation** කියන්නේ හරියට සිනමා ශාලාවක එක පේළියක අසුන් 10ක් එකවර වෙන් කරගන්නවා වගේ වැඩක්. ඒක ලේසියි, වේගවත්. හැබැයි ඒ පේළියේ මැද අසුනක් දැනටමත් වෙන කෙනෙක් අරන් තිබුණොත් ඔයාට ඒ පේළියම ගන්න බැරි වෙනවා (**Fragmentation**). ඒ ගැටලුව විසඳන්න තමයි **Extents** පාවිච්චි කරන්නේ.

**I/O operations** (Input/Output operations) කියන එක සරලවම කිව්වොත්, Operating System එක විසින් Hard Disk එකෙන් හෝ SSD එකෙන් දත්ත ලබා ගැනීම (**Read**) හෝ එහි දත්ත ගබඩා කිරීම (**Write**) සඳහා කරන එක් "ගනුදෙනුවක්" (Transaction) ලෙස හඳුන්වන්න පුළුවන්.


**Linked Allocation** කියන්නේ Hard Disk එකේ දත්ත ගබඩා කරන තවත් එක් ක්‍රමයක්. මෙය හරියට "දාමයක්" (Chain) වගේ ක්‍රියාත්මක වන ක්‍රමයක්. අපි මෙහි ප්‍රධාන කරුණු සරලව තේරුම් ගනිමු.

### 1. වැඩ කරන ආකාරය (How it works)

- **Linked List of Blocks:** මෙහිදී සෑම File එකක්ම එකිනෙකට සම්බන්ධ වූ Blocks සමූහයකින් සමන්විතයි.
    
- **Pointers:** සෑම Block එකක් ඇතුළේම **දත්ත (Data)** වලට අමතරව, ඊළඟ Block එක තියෙන තැන පෙන්වන **Pointer** එකක් (ලිපිනයක්) අඩංගු වෙනවා.
    
- **Nil Pointer:** File එකේ අවසාන Block එකේ තියෙන Pointer එක "Nil" හෝ -1 ලෙස සටහන් වෙනවා. ඒ කියන්නේ එතනින් File එක ඉවරයි කියන එකයි.
    

---

### 2. මෙහි ඇති වාසි (Advantages)

- **No External Fragmentation:** මේ ක්‍රමයේදී දත්ත එක පෙළට (**Contiguous**) තියෙන්න අවශ්‍ය නැහැ. Hard Disk එකේ ඕනෑම තැනක තියෙන හිස් Block එකක් පාවිච්චි කරන්න පුළුවන්. ඒ නිසා ඉඩ අපතේ යන්නේ නැහැ.
    
- **No Compaction:** ඉඩ අපතේ නොයන නිසා, තැටියේ තියෙන Files එහා මෙහා කරලා ඉඩ හදන ක්‍රියාවලිය (**Compaction**) මෙයට අවශ්‍ය වෙන්නේ නැහැ.
    
- **Easy Growth:** File එකක් ලොකු කරන්න ඕන වුණොත්, පද්ධතිය කරන්නේ ඕනෑම තැනක තියෙන හිස් block එකක් සොයාගෙන (**Free space management**) එය අවසාන block එකට සම්බන්ධ (Link) කරන එක විතරයි.
    

---

### 3. මෙහි ඇති අවාසි සහ ගැටලු (Disadvantages)

- **Locating is Slow (I/O Overhead):** අපි කලින් කතා කළා වගේ, මේ ක්‍රමයේදී **Direct Access** කරන්න බැහැ. 10 වෙනි Block එකට යන්න නම්, 1 ඉඳන් 9 වෙනි එක වෙනකම් පිළිවෙලින් කියවාගෙන යන්නම ඕනේ. මේ නිසා විශාල **I/O operations** ප්‍රමාණයක් සහ **Disk seeks** (තැටියේ හිස එහා මෙහා වීම්) ප්‍රමාණයක් වැය වෙනවා.
    
- **Reliability (විශ්වාසනීයත්වය):** මේක ලොකු ප්‍රශ්නයක්. යම් හෙයකින් එක block එකක තියෙන **Pointer** එකක් "Corrupt" වුණොත් (නැති වුණොත්), ඒකෙන් පස්සේ තියෙන මුළු File එකම පාවිච්චි කරන්න බැරි වෙනවා.
    
- **Space for Pointers:** සෑම Block එකක් ඇතුළෙම Pointer එකක් තබන්න ඉඩ වෙන් කරන්න වෙනවා. උදාහරණයක් ලෙස Block එක 512 bytes නම්, එයින් bytes 4ක් pointer එකට වෙන් කළොත් දත්ත ගබඩා කරන්න ඉතුරු වෙන්නේ 508 bytes විතරයි.
    

---

### 4. කාර්යක්ෂමතාව වැඩි කිරීම (Optimization)

මේ ක්‍රමයේ වේගය වැඩි කරන්න **Clustering** කියන ක්‍රමය පාවිච්චි කරනවා.

- **Clustering යනු:** Blocks කිහිපයක් (උදා: 4ක්) එකතු කරලා එක "කාණ්ඩයක්" (Group) ලෙස සැලකීමයි.
    
- **ප්‍රතිඵලය:** එවිට Pointers ගණන අඩු වී වේගය වැඩි වෙනවා.
    
- **ගැටලුව:** හැබැයි මෙහිදී **Internal Fragmentation** (කුඩා file එකක් ගබඩා කිරීමේදී cluster එකක් ඇතුළේ ඉඩ අපතේ යාම) වැඩි වෙනවා.
    

**සාරාංශය:** Linked Allocation කියන්නේ හරියට "සැඟවුණු අරමුදල් සොයන" (Treasure Hunt) සෙල්ලමක් වගෙයි. එක තැනක තියෙන ඉඟියෙන් තමයි ඊළඟ තැන තියෙන්නේ කොහේද කියලා කියන්නේ. මේක ඉඩ පාවිච්චි කරන්න හොඳ ක්‍රමයක් වුණත්, වේගය අතින් ගොඩක් මන්දගාමීයි.


**Linked Allocation** ක්‍රමයේදී දත්ත ලබාගැනීමට වැයවන **I/O operations** ගණනය කරන ආකාරය ඔයා දීපු උදාහරණ ඇසුරින් සරලව තේරුම් ගනිමු.

මෙහි ඇති ප්‍රධානතම ලක්ෂණය වන්නේ, එක් block එකක් ඇතුළේ ඊළඟ block එක තිබෙන ලිපිනය (**Pointer**) ගබඩා කර තිබීමයි.

### 1. 6 වෙනි Block එක ලබා ගැනීමට 6 I/O වැය වෙන්නේ ඇයි?

හිතන්න ඔයාට 10ක් තියෙන file එකක **6 වෙනි block** එක කියවන්න ඕන කියලා.

- **ගැටලුව:** Operating System එක දන්නේ **1 වෙනි block** එක තියෙන තැන විතරයි. 2 වෙනි එක කොහේද තියෙන්නේ කියලා තියෙන්නේ 1 වෙනි එක ඇතුළේ. 3 වෙනි එක තියෙන තැන තියෙන්නේ 2 ඇතුළේ.
    
- **පියවර:**
    
    1. 1 වෙනි block එක කියවනවා (1 I/O) -> දැන් 2 වෙනි එකේ ලිපිනය දන්නවා.
        
    2. 2 වෙනි block එක කියවනවා (2 I/O) -> දැන් 3 වෙනි එකේ ලිපිනය දන්නවා.
        
    3. 3 වෙනි block එක කියවනවා (3 I/O) -> දැන් 4 වෙනි එකේ ලිපිනය දන්නවා.
        
    4. 4 වෙනි block එක කියවනවා (4 I/O) -> දැන් 5 වෙනි එකේ ලිපිනය දන්නවා.
        
    5. 5 වෙනි block එක කියවනවා (5 I/O) -> දැන් 6 වෙනි එකේ ලිපිනය දන්නවා.
        
    6. **අවසානයේ 6 වෙනි block එක කියවනවා (6 I/O).**
        
- **නිගමනය:** අපිට අවශ්‍ය block එකට කෙලින්ම පනින්න බැහැ (**No Direct Access**). ඒකට යන පාර හොයාගන්න කලින් තියෙන ඔක්කොම blocks කියවන්න වෙනවා (**Traverse**). ඒ නිසයි **6 I/O operations** අවශ්‍ය වෙන්නේ.

**FAT (File Allocation Table)** කියන්නේ දත්ත ගබඩා කරන ඉතාමත් ජනප්‍රිය ක්‍රමයක්. අපි කලින් කතා කරපු **Linked Allocation** ක්‍රමයේ තිබුණ ප්‍රධානතම අඩුපාඩුවක් වන මන්දගාමී බව මඟහරවා ගන්න තමයි මේ ක්‍රමය හඳුන්වා දුන්නේ.

අපි මේක සරල කරුණු කිහිපයකින් තේරුම් ගනිමු:

### 1. මේක ක්‍රියාත්මක වෙන්නේ කොහොමද?

Linked Allocation වලදී pointers තිබුණේ ඇත්තම Data Blocks ඇතුළේය. හැබැයි **FAT** ක්‍රමයේදී කරන්නේ ඒ pointers ඔක්කොම ඒ ඒ blocks වලින් ඉවත් කරලා, ඒවා එකතු කරලා වෙනම වගුවක් (**Table**) හදන එකයි.

- මේ වගුව (FAT Table) Hard Disk එකේ මුලම (Beginning of the volume) ගබඩා කරලා තියෙනවා.
    
- වගුවේ සෑම කොටසකම (Entry) ලියා තිබෙන්නේ ඊළඟට කියවිය යුතු Block එකේ අංකයයි.
    
- File එකේ අවසානය පෙන්වීමට **EOF (End of File)** වැනි විශේෂ අගයක් පාවිච්චි කරනවා.
    

---

### 2. ඇයි FAT ක්‍රමය වේගවත් වෙන්නේ?

මෙම ක්‍රමයේ ඇති ලොකුම වාසිය තමයි **Caching**.

- **RAM එකට ගැනීම:** Operating System එක සාමාන්‍යයෙන් කරන්නේ Disk එකේ තියෙන මේ **FAT Table** එක කියවලා ඒක පරිගණකයේ **RAM** එකේ තබා ගැනීමයි.
    
- **සොයාගැනීම:** දැන් අපිට යම් File එකක මැද තියෙන දත්තයක් අවශ්‍ය වුණොත්, OS එකට පුළුවන් RAM එකේ තියෙන Table එක බලලා ඒ දත්තය තියෙන තැන ක්ෂණයකින් හොයාගන්න. Linked Allocation වල වගේ Disk එකේ blocks එකින් එක පීරන්න (Traverse කරන්න) අවශ්‍ය වෙන්නේ නැහැ.
    

---

### 3. I/O Operations ගණනය කිරීම ($Q + R$)

අපි කලින් කතා කරපු පින්තූරයේ තිබුණ ගණනය කිරීම මෙහිදී වැදගත් වෙනවා:

- **Case 1 (FAT is in RAM):** Table එක RAM එකේ තියෙන නිසා ඒක බලන්න Disk එකට යන්න ඕන නැහැ (Q=0). කෙලින්ම දත්තය තියෙන තැනට ගිහින් ඒක කියවනවා (R=1). මුළු එකතුව **1 I/O**.
    
- **Case 2 (FAT is in Disk):** Table එක Disk එකේ නම්, මුලින්ම Table එක කියවන්න Disk එකට යන්න ඕනේ (Q=1). ඊට පස්සේ දත්තය ගන්න Disk එකට යන්න ඕනේ (R=1). මුළු එකතුව **2 I/O operations**.

### 4. වාසි සහ අවාසි

**වාසි:**

- **Random Access / Direct Access** සඳහා සහාය දක්වන නිසා වේගවත්ය.
    
- **No External Fragmentation:** Blocks එක පෙළට තිබීම අනිවාර්ය නැත.
    
- USB Pen Drives සහ SD Cards සඳහා ඉතා සුදුසුයි.
    

**අවාසි:**

- **Space Overhead:** මේ Table එක ගබඩා කරන්න Hard Disk එකේ වෙනම ඉඩක් වෙන් කරන්න වෙනවා.
    
- **Reliability:** FAT Table එකට හානි වුණොත් මුළු Disk එකේම තියෙන දත්ත සොයාගැනීමට නොහැකි වෙනවා. (ඒ නිසා සාමාන්‍යයෙන් මෙහි පිටපත් දෙකක් පවත්වාගෙන යනවා).

**Indexed Allocation Method** කියන්නේ Operating System එකක් Hard Disk එකේ දත්ත ගබඩා කිරීමේදී පාවිච්චි කරන ඉතාමත් ක්‍රමවත් සහ දියුණු ක්‍රමයක්.

මෙය සරලව තේරුම් ගන්න අපි මෙහෙම හිතමු:

ඔයා ගාව විශාල පොතක් තියෙනවා, හැබැයි ඒ පොතේ පරිච්ඡේද (Chapters) තියෙන්නේ පොතේ එක එක තැන්වල විසිරිලා. ඔයාට ඒ පරිච්ඡේද ඉක්මනින් හොයාගන්න නම් මොකක්ද කරන්නේ? ඔයා පොතේ මුල තියෙන **පටුන (Index)** බලනවා. අන්න ඒ වගේ තමයි මේ ක්‍රමයත් වැඩ කරන්නේ.

### 1. මේක වැඩ කරන්නේ කොහොමද? (How it works?)

- **Index Block:** මෙහිදී සෑම File එකක් සඳහාම වෙනම **Index Block** එකක් (වෙනම block එකක්) වෙන් කරනවා.
    
- **Pointers:** මේ Index Block එක ඇතුළේ අදාළ File එකට අයිති අනෙක් සියලුම දත්ත Blocks (Data blocks) තියෙන ලිපින (Addresses) පිළිවෙලට ලැයිස්තුගත කරලා තියෙනවා.
    
- **Directory Entry:** File System එකේ ඩිරෙක්ටරිය ඇතුළේ සටහන් වෙන්නේ අර **Index Block** එක පිහිටා ඇති ස්ථානය විතරයි.
    

---

### 2. කලින් ක්‍රම සමඟ මෙහි ඇති සම්බන්ධය (Connection to previous methods)

අපි කලින් කතා කරපු ක්‍රමවල තිබුණ ප්‍රධාන අඩුපාඩු මේ ක්‍රමය මගින් විසඳනවා:

1. **Contiguous Allocation සමඟ:**
    
    - **ප්‍රශ්නය:** Contiguous වලදී එක පෙළට ඉඩ තිබිය යුතුයි (**External Fragmentation**).
        
    - **විසඳුම:** Indexed ක්‍රමයේදී දත්ත Blocks තැටියේ ඕනෑම තැනක විසිරී තිබිය හැකියි. ඒ නිසා ඉඩ අපතේ යන්නේ නැහැ.
        
2. **Linked Allocation සමඟ:**
    
    - **ප්‍රශ්නය:** Linked ක්‍රමයේදී මැද තියෙන දත්තයක් බලන්න නම් මුල ඉඳන් පීරන්න ඕනේ (**No Direct Access**).
        
    - **විසඳුම:** Indexed ක්‍රමයේදී Index Block එක බැලූ සැණින් ඕනෑම Block එකක ලිපිනය දන්නා නිසා කෙලින්ම ඒ තැනට පනින්න පුළුවන් (**Direct Access**).
        
3. **FAT (File Allocation Table) සමඟ:**
    
    - **සම්බන්ධය:** FAT කියන්නේ මුළු Hard Disk එකටම තියෙන එකම "පොදු පටුනක්" (Global table) වගේ දෙයක්. හැබැයි Indexed Allocation වලදී වෙන්නේ සෑම File එකකටම තමන්ගේම කියලා "පුද්ගලික පටුනක්" (Local index) තිබීමයි.

### 3. I/O Operations ගණනය කිරීම

හිතන්න අපිට File එකක **8 වෙනි Block** එක කියවන්න ඕන කියලා:

1. මුලින්ම **Index Block** එක කියවනවා (1 I/O) - දැන් අපි 8 වෙනි block එක තියෙන තැන දන්නවා.
    
2. ඊට පස්සේ කෙලින්ම **8 වෙනි Block** එක කියවනවා (තවත් 1 I/O).
    

- **මුළු එකතුව:** 2 I/O operations. (Linked allocation වලදී වගේ blocks 7ක්ම කියවාගෙන යන්න අවශ්‍ය වෙන්නේ නැහැ).
    

---

### 4. විශාල ගොනු සඳහා (Handling Large Files)

සමහර වෙලාවට File එකක් ගොඩක් ලොකු වුණොත් එක Index Block එකක් ඇතුළේ ලිපින ඔක්කොම ලියන්න ඉඩ මදි වෙන්න පුළුවන්. ඒ වෙලාවට පද්ධතිය මේ වගේ දේවල් කරනවා:

- **Linked Scheme:** එක index block එකක් අවසානයේ තවත් index block එකක ලිපිනයක් දීම.
    
- **Multilevel Index:** Index block එකක් ඇතුළේ තවත් index blocks වල ලිපින තැබීම (හරියට පටුනක් ඇතුළේ තවත් පටුනක් තියෙනවා වගේ).
    

**වාසි:**

- **Direct access** ලබා දීම.
    
- **Fragmentation** ගැටලුව නැති කිරීම.
    

**අවාසි:**

- කුඩා Files සඳහාත් මුළු Index Block එකක්ම වෙන් කරන්න වෙන නිසා ඉඩ ටිකක් අපතේ යාම (**Index overhead**).


ඔයාගේ ප්‍රශ්නය හරිම තාර්කිකයි. ඔව්, **FAT (File Allocation Table)** ක්‍රමයේදී මුළු **Partition** (උදාහරණයක් ලෙස C: හෝ D: Drive එක) එකේම තියෙන **සියලුම ගොනු (Files)** වල දත්ත Blocks එකිනෙකට සම්බන්ධ වන ආකාරය පෙන්වන "සිතියම" තියෙන්නේ මේ එකම Table එක ඇතුළෙයි.

අපි මේ වෙනස තවත් පැහැදිලිව බලමු:

### 1. FAT (File Allocation Table) - "පොදු පටුන"

FAT එක ක්‍රියාත්මක වෙන්නේ මුළු Disk එකටම (හෝ Partition එකකට) එක මධ්‍යම වගුවක් විදියටයි.

- **විස්තර:** මේ Table එකේ තියෙන්නේ මුළු Disk එකේම තියෙන **සෑම Block එකකටම අදාළ Entry එකක්**.
    
- **Details:** යම් File එකක් Blocks කිහිපයක විසිරිලා තියෙනවා නම්, 1 වෙනි block එකෙන් 2 වෙනි එකටත්, 2න් 3ටත් යන "පාර" මේ එකම වගුව ඇතුළේ සටහන් වෙලා තියෙනවා.
    
- **ඔක්කොම තියෙනවද?** ඔව්, ඒ Partition එකේ තියෙන Files 1000ක් වුණත්, ඒ 1000ම Blocks වල සම්බන්ධතාවය (Chaining) තියෙන්නේ මේ **එකම FAT Table** එකේයි.
    

---

### 2. Indexed Allocation - "පුද්ගලික පටුන"

මෙතැනදී මුළු Computer එකටම හෝ Disk එකටම කියලා එක පොදු Table එකක් නැහැ.

- **විස්තර:** සෑම File එකක්ම නිර්මාණය කරන විට, ඒ File එකටම වෙන් වුණු **වෙනම Block එකක්** (Index Block) පද්ධතිය මගින් ලබා දෙනවා.
    
- **Details:** අන්න ඒ Index Block එක ඇතුළේ තියෙන්නේ **ඒ අදාළ File එකේ දත්ත තියෙන තැන් විතරයි**. වෙනත් Files වල විස්තර ඒක ඇතුළේ නැහැ.
    
- **වෙනස:** මෙහිදී Files 1000ක් තියෙනවා නම්, Index Blocks 1000ක් පද්ධතියේ තියෙනවා.
    

---

### සරල උදාහරණයකින් (Analogy):

- **FAT (Master Register):** හිතන්න ලොකු **ඉස්කෝලෙක තියෙන ප්‍රධාන ලේඛනාගාරය** වගේ කියලා. ඒකේ එකම පොතක ඉස්කෝලේ ඉන්න හැම ළමයෙක්ගෙම (Files) ලිපින සහ විස්තර ලියලා තියෙනවා. කාගේ විස්තරයක් ඕන වුණත් හැමෝම යන්නේ ඒ එකම පොත ගාවටයි.
    
- **Indexed (Personal Diary):** හිතන්න **හැම ළමයෙක්ම තමන්ගේම ඩයරි එකක්** පාවිච්චි කරනවා වගේ කියලා. ඒ ළමයාගේ ලිපිනය සහ විස්තර තියෙන්නේ ඒ ළමයා ගාව තියෙන ඩයරි එකේ විතරයි. ඉස්කෝලේ ළමයි 500ක් හිටියොත් ඩයරි 500ක් තියෙනවා.
    

### මෙහි ප්‍රායෝගික වෙනස:

- **FAT වලදී:** OS එකට Disk එකේ තියෙන ඕනෑම File එකක් කොහේද තියෙන්නේ කියලා බලන්න අර එකම FAT Table එක පාවිච්චි කරන්න පුළුවන්. ඒක RAM එකට දාගත්තම වැඩේ ලේසියි.
    
- **Indexed වලදී:** OS එක මුලින්ම බලන්න ඕනේ අදාළ File එකේ "Index Block" එක තියෙන්නේ කොහේද කියලා. ඒක හොයාගත්තට පස්සේ තමයි File එකේ දත්ත ටික කියවන්න පුළුවන් වෙන්නේ.
    

දැන් අර "පොදු පටුන" (FAT) සහ "පුද්ගලික පටුන" (Indexed) අතර වෙනස පැහැදිලිද? FAT කියන්නේ ඇත්තටම මුළු Drive එකටම තියෙන එකම Map එකයි.


ඔයා දීලා තියෙන සටහනෙන් පැහැදිලි කරන්නේ විවිධ **File Allocation Methods** වලදී දත්ත ලබාගැනීමට වැයවන **I/O Operations** ගණනය කරන ආකාරයයි. අපි මේක ක්‍රම තුනටම අදාළව සරලව තේරුම් ගනිමු.

හිතන්න අපිට **10**ක් blocks තියෙන File එකක **7 වෙනි block** එක කියවන්න අවශ්‍යයි කියලා.

---

### 1. Contiguous Allocation (සන්තතිකව වෙන් කිරීම)

මෙහිදී blocks සියල්ලම තැටියේ එක පෙළට පිහිටා ඇත.

- **වැඩ කරන හැටි:** OS එක දන්නවා File එක පටන් ගන්නා තැන. ඒ නිසා 7 වෙනි එක තියෙන තැන සූත්‍රයකින් ගණනය කරලා, වෙන කොහේවත් බලන්නේ නැතුව කෙලින්ම ඒ තැනට පනින්න පුළුවන් (**Direct Access**).
    
- **I/O ගණන:** **1 I/O** (අදාළ data block එක කියවීමට පමණි).
    

---

### 2. Linked Allocation (සම්බන්ධිතව වෙන් කිරීම)

මෙහිදී blocks විසිරී පවතින අතර එකක ලිපිනය තියෙන්නේ ඊට කලින් එක ඇතුළෙයි.

- **වැඩ කරන හැටි:** අපිට 7 වෙනි එකට යන්න නම්, අනිවාර්යයෙන්ම 1 වෙනි එකේ ඉඳන් 2, 3, 4, 5, 6 කියන blocks ඔක්කොම පිළිවෙලින් කියවාගෙන යන්නම වෙනවා (**Sequential Access**).
    
- **I/O ගණන:** **7 I/Os** (සෑම block එකක්ම පීරීමට සිදු වන නිසා).
    

---

### 3. Indexed Allocation (දර්ශක මගින් වෙන් කිරීම)

මෙහිදී සියලුම blocks වල ලිපින එකම **Index Block** එකක සටහන් කර ඇත.

- **වැඩ කරන හැටි:** 1. මුලින්ම **Index Block** එක කියවනවා (1 I/O). දැන් අපි දන්නවා 7 වෙනි එක තියෙන තැන.
    
    2. ඊට පස්සේ කෙලින්ම **7 වෙනි Data Block** එකට ගිහින් ඒක කියවනවා (1 I/O).
    
- **I/O ගණන:** 1 + 1 = **2 I/Os**.
    

**විශේෂ අවස්ථාව:** Index block එක දැනටමත් පරිගණකයේ **RAM (Memory)** එකේ තියෙනවා නම්, Disk එකේ තියෙන Index block එක කියවන්න අවශ්‍ය වෙන්නේ නැහැ. එවිට අවශ්‍ය වෙන්නේ ඇත්තම දත්තය කියවීමට යන **1 I/O** එක පමණයි.

### සම්පූර්ණ File එකම (Blocks 10ක්) කියවීමට යන I/O ගණන:

මෙයත් විභාගවලට නිතරම අහන දෙයක්.

- **Contiguous:** 10 I/Os.
    
- **Linked:** 10 I/Os.
    
- **Indexed:** **11 I/Os** (Index Block එක සඳහා 1ක් සහ Data Blocks 10 සඳහා 10ක්).
    

**සාරාංශය (Comparison Table):**

| **Method**     | **Accessing 7th Block** | **Reading Entire File (10 blocks)** |
| -------------- | ----------------------- | ----------------------------------- |
| **Contiguous** | 1 I/O                   | 10 I/Os                             |
| **Linked**     | 7 I/Os                  | 10 I/Os                             |
| **Indexed**    | 2 I/Os                  | 11 I/Os                             |
අපි මේ කොටස සරල කරුණු කිහිපයකින් තේරුම් ගනිමු. මෙතැනදී ප්‍රධාන වශයෙන් කතා කරන්නේ නවීන **File Systems** වල ස්වභාවය සහ ඒවායේ වේගය (**Performance**) තීරණය වන සාධක ගැනයි.

### 1. Modern File Systems (නවීන ගොනු පද්ධති)

නවීන පද්ධති නිර්මාණය කරලා තියෙන්නේ ඉතාමත් ඉහළ විශ්වාසනීයත්වයක් (**Reliability**) ඇතිව සහ ඉතා විශාල ධාරිතාවන් සහිත දත්ත ගබඩා (**Large storages**) හැසිරවිය හැකි ආකාරයටයි.

**ප්‍රධාන උදාහරණ:**

- **NTFS (New Technology File System):** Windows මෙහෙයුම් පද්ධති වල ප්‍රධාන වශයෙන් භාවිතා වේ.
    
- **ext4 (fourth extended filesystem):** Linux පද්ධති වල බහුලවම භාවිතා වන ක්‍රමයයි.
    
- **APFS (Apple File System):** macOS සහ iOS (Apple උපාංග) සඳහා විශේෂයෙන් නිර්මාණය කර ඇත.
    

---

### 2. File System Performance (ගොනු පද්ධතියක වේගය)

ගොනු පද්ධතියක් කොතරම් වේගයෙන් වැඩ කරනවාද යන්න ප්‍රධාන සාධක 3ක් මත රඳා පවතී:

1. **Disk speed:** මෙය දෘඩාංග (**Hardware**) මත පදනම් වේ. උදාහරණයක් ලෙස සාමාන්‍ය HDD එකකට වඩා **SSD** එකක වේගය ඉතා වැඩිය.
    
2. **Allocation method:** අපි කලින් ඉගෙන ගත්තු දත්ත ගබඩා කරන ක්‍රමය (Contiguous, Linked හෝ Indexed) මෙයට බලපායි. Contiguous ක්‍රමය වේගවත් වන අතර Linked ක්‍රමය මන්දගාමී වේ.
    
3. **Caching mechanisms:** දත්ත කියවන වාරයක් පාසා මන්දගාමී Disk එකට යන්නේ නැතිව, නිතර භාවිතා වන දත්ත වේගවත් මතකයක (**RAM**) තබා ගැනීමේ ක්‍රමයයි.
    

---

### 3. Buffer Cache (බෆර් කෑෂ්)

නවීන Operating Systems වල කාර්යක්ෂමතාව (**Efficiency**) වැඩි කිරීමට පාවිච්චි කරන ප්‍රධානම උපක්‍රමය මෙයයි.

- **ක්‍රියාවලිය:** පද්ධතිය විසින් දත්ත කියවන විට හෝ ලියන විට, ඒවා කෙලින්ම Hard Disk එකට යවන්නේ නැතිව **RAM** එකේ ඇති **Buffer cache** නම් විශේෂ කොටසක තාවකාලිකව තබා ගනී.
    
- **වාසිය:** ඊළඟ වතාවේ එම දත්තයම අවශ්‍ය වුණොත්, මන්දගාමී Disk එකට යන්නේ නැතිව RAM එකෙන් ඉතා ඉක්මනින් ලබා ගත හැකියි. මෙයින් **Disk I/O operations** ගණන අඩු වී මුළු පද්ධතියේම වේගය වැඩි වේ.
    

**සාරාංශය:** නවීන File Systems (NTFS, ext4) ඉතා දියුණුයි. ඒවායේ වේගය වැඩි කරන්න ප්‍රධාන වශයෙන්ම දෘඩාංග වල වේගය, දත්ත අසුරන ක්‍රමය සහ **Buffer caching** කියන ක්‍රමවේදයන් උදවු වෙනවා.

දැන් ඔයාට මේ කරුණු ටික පැහැදිලිද? මින් ඉදිරියට අපි **Disk Scheduling** (Disk එකේ දත්ත කියවන පිළිවෙල පාලනය කරන හැටි) ගැන කතා කරමුද?
